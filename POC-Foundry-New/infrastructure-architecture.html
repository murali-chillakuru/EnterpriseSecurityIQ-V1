1. Solution Overview

EnterpriseSecurityIQ is an AI-powered compliance assessment and tenant intelligence agent for Microsoft Azure and Entra ID environments. It runs as a containerized Python (FastAPI) application hosted on Azure Container Apps, leveraging Azure OpenAI (GPT-5.1) for natural-language orchestration of 13 security assessment tools.

📐 Architecture Style

Follows the Microsoft Azure Well-Architected Framework (WAF) pillars: Reliability, Security, Cost Optimization, Operational Excellence, and Performance Efficiency. The deployment uses a serverless microservice pattern — single container workload with managed infrastructure, user-delegated authentication, and managed identity for service-to-service calls.

4. Identity & Authentication Architecture

The system uses a dual-credential model — user-delegated tokens for tenant assessment operations and managed identity for infrastructure operations.

4.1 User Authentication Flow (MSAL.js SPA)

1
SPA Login: User navigates to the dashboard. MSAL.js initiates OAuth 2.0 Authorization Code flow with PKCE against Microsoft Entra ID. Supports "Switch Tenant" via authority override.
2
Token Acquisition: SPA acquires two tokens silently — Graph token (https://graph.microsoft.com/.default) for Entra/M365 queries and ARM token (https://management.azure.com/.default) for Azure resource queries.
3
API Call: SPA sends POST /chat with {message, graph_token, arm_token}. The backend decodes the ARM JWT's tid claim to identify the user's tenant (v39-tenant fix).
4
Token Wrapping: Backend wraps tokens into UserTokenCredential (implements AsyncTokenCredential), which routes to Graph or ARM token based on requested scopes.
5
Assessment Execution: All ARM API, Graph API, and Azure Resource Graph calls use the user's delegated tokens — the assessment runs in the context of the user's tenant, not the hosting tenant.

✅ Security Design Principle

No secrets are stored anywhere — the managed identity provides passwordless authentication to all Azure services. ACR uses MI (not admin user), Storage uses MI (shared key disabled), Key Vault uses MI + RBAC (not access policies). The SPA does not handle any secrets — only short-lived OAuth tokens via PKCE.

5. Data Flow Architecture

5.1 Assessment Request Flow

1
User → SPA: User types a request (e.g. "Run compliance assessment") in the chat interface.
2
SPA → Container App: POST /chat with SSE streaming. Includes user's Graph + ARM tokens and conversation history.
3
Container App → Azure OpenAI: Sends system prompt + user message + tool definitions. OpenAI returns tool_calls (e.g. run_assessment).
4
Container App → User's Tenant: Executes tool functions — calls ARM API (via user's ARM token) to collect Azure resources, Graph API (via user's Graph token) to collect Entra data.
5
Container App (internal): Evidence → Evaluation Engine → Compliance scoring → Report generation (HTML, JSON, PDF, Excel).
6
Container App → Storage: Uploads generated reports to blob storage (managed identity). Returns SAS-free download URLs via the /reports/ API.
7
Container App → SPA: Streams SSE events — tool status updates, progress percentages, and final markdown response with report links.

⚠️ Network Gap

The Container Apps Environment is deployed without VNet integration. All service-to-service calls (Container App → OpenAI, Storage, ACR) traverse public endpoints. While RBAC and Entra ID authentication provide strong identity-layer security, there is no network-layer defense in depth. The application endpoint is publicly accessible without IP restrictions or WAF.

9.2 Recovery Procedure

  • Container App failure: Platform auto-restarts. If persistent, redeploy from ACR image tag.
  • Region failure: Rebuild Container Apps Environment in alternate region, point to same ACR and AI Services.
  • Data recovery: All assessment data is ephemeral (in-memory). Reports in Storage have LRS replication.
