Authenticate to Azure

✅ Unified Credential The engine uses a single DefaultAzureCredential for both ARM and Graph — no separate consent flows, no browser popups, no Connect-MgGraph. 
# Interactive
az login --tenant "4a3eb5f4-1ec6-4a73-bb03-1ca63cb52d67"

# Service Principal (CI/CD)
export AZURE_TENANT_ID="4a3eb5f4-..."
export AZURE_CLIENT_ID="<client-id>"
export AZURE_CLIENT_SECRET="<secret>"

# App Registration (dedicated)
export ENTERPRISESECURITYIQ_APP_CLIENT_ID="<app-id>"
export ENTERPRISESECURITYIQ_APP_CLIENT_SECRET="<app-secret>"
export ENTERPRISESECURITYIQ_AUTH_MODE="appregistration"

🚀 Quick Start

cd AIAgent

# Interactive — prompts for tenant ID and framework selection
python run_assessment.py

# Non-interactive
python run_assessment.py --tenant "4a3eb5f4-..." --framework FedRAMP

Assessment Pipeline

sequenceDiagram participant U as User / CLI participant A as run_assessment.py participant C as ComplianceCredentials participant O as Orchestrator participant Col as 64 Collectors participant Ev as Evaluators participant R as Report Generators U->>A: python run_assessment.py --tenant ... --framework FedRAMP A->>C: authenticate(tenant_id, auth_mode) C->>C: DefaultAzureCredential (ARM + Graph) C->>C: preflight_check() C-->>A: credentials + permission status A->>O: run(config, credentials) O->>Col: discover_collectors() → 68 functions par Azure Collectors (batches) Col->>Col: 47 Azure collectors (P10-P240) and Entra Collectors (batches) Col->>Col: 17 Entra collectors + 2 standalone end Col-->>O: raw evidence O->>Ev: evaluate(evidence, framework_mappings) Ev->>Ev: ~120 check functions × 10 domains Ev-->>O: findings + control results O->>R: generate_reports(findings, results) R-->>O: HTML + MD + Excel + JSON O-->>A: output directory path A-->>U: Summary + compliance score
How it works:
  1. Authenticate — ComplianceCredentials obtains DefaultAzureCredential for both ARM and Graph (no popup), then runs a preflight check probing 3 Graph endpoints.
  2. Discover — The orchestrator calls discover_collectors() which scans the registry of @register_collector-decorated functions, returning 68 collector functions sorted by priority.
  3. Collect — Azure and Entra collector groups run concurrently. Within each group, collectors execute in configurable batch sizes (default: 8 Azure, 6 Entra).
  4. Evaluate — ~120 evaluation functions across 10 domains analyze the collected evidence. Cross-domain fallback (28 routes) fills gaps. Suppressions are applied.
  5. Report — Per-framework subfolders get HTML, Markdown, and Excel reports. JSON exports (findings, control results, evidence) go to the root output directory.

🔐 Authentication

Handled by ComplianceCredentials (app/auth.py) with 4 modes. Both ARM and Graph use the same DefaultAzureCredential.

Managed Identity Azure-hosted

When running in Azure (Container Apps, App Service, VM), managed identity is used automatically. No env vars needed.

Preflight Validation Details

Before collection starts, preflight_check() probes 3 Graph endpoints:

ProbeEndpointPermissionOn 403On 401
Users/users?$top=1User.Read.AllWarningBlocking
Conditional Access/identity/conditionalAccess/policiesPolicy.Read.AllWarningBlocking
Role Management/roleManagement/directory/roleAssignmentsRoleManagement.Read.AllWarningBlocking

403 = assessment continues with reduced scope (missing evidence). 401 = assessment stops with actionable error.

🎯 Selective Assessment Patterns

Azure-Only (No Entra)

{
  "collectors": {
    "azureEnabled": true,
    "entraEnabled": false
  }
}

Entra-Only (No Azure)

{
  "collectors": {
    "azureEnabled": false,
    "entraEnabled": true
  }
}

Subscription Filter

{
  "auth": {
    "subscriptionFilter": [
      "d33fc1a7-xxxx-...",
      "0eb177bd-xxxx-..."
    ]
  }
}

Or: export AZURE_SUBSCRIPTION_FILTER="id1,id2"

Large Tenant (50K+ Users)

{
  "collectors": {
    "userSampleLimit": 5000,
    "azureBatchSize": 6,
    "entraBatchSize": 4,
    "collectorTimeout": 600
  }
}

Multi-Tenant Assessment

python run_assessment.py --tenant "primary" \
  --additional-tenants "tenant-2" "tenant-3" --framework FedRAMP

Or set "additionalTenants": ["tenant-2", "tenant-3"] in config JSON. Each tenant's evidence is collected separately, then combined for cross-tenant evaluation.

🔧 Troubleshooting

Check Authentication 
# Verify Azure CLI login
az account show --query '{tenant:tenantId, subscription:name}'

# List accessible subscriptions
az account list --query '[].{name:name, id:id, state:state}' -o table
Assessment Runs Slow

Large tenants may take 10+ minutes. To speed up:

{
  "collectors": {
    "userSampleLimit": 5000,
    "azureBatchSize": 6,
    "entraBatchSize": 4,
    "collectorTimeout": 120
  }
}
Sign-In Logs Return 0 Records

Expected if the tenant lacks Entra ID P1/P2 license. The assessment continues with missing_evidence status for sign-in log controls.

Collector Timeout

If collectors timeout on large subscriptions:

{ "collectors": { "collectorTimeout": 600 } }
Permission Errors

Assessment continues despite permission failures. Check access-denied.json in the output for details. The methodology report also lists permission gaps.

Resume After Failure

With checkpointEnabled: true (default), the assessment saves progress to .checkpoint.json. Re-run the same command and it resumes from the last checkpoint. The checkpoint is auto-removed on success.

Missing Evidence Types

If controls show missing_evidence, check:

  1. Permissions — verify access-denied.json for 403 errors
  2. Collector toggles — ensure azureEnabled/entraEnabled are true
  3. Resource existence — some evidence requires specific Azure resources (e.g., Sentinel, Purview)
