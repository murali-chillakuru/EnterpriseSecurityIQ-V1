sequenceDiagram participant U as User / Agent Tool participant O as postureiq_orchestrator.py participant R as Registry participant AZ as Azure Collectors (50) participant EN as Entra Collectors (18) participant EV as engine.py participant RP as Report Generators (29) U->>O: run_postureiq_assessment(creds, config) O->>R: discover_collectors() R-->>O: sorted by priority par Azure Group (batch_size=8) O->>AZ: Concurrent batches AZ-->>O: evidence[] and Entra Group (batch_size=6) O->>EN: Concurrent batches EN-->>O: evidence[] end Note over O: Checkpoint saved O->>EV: evaluate_all(evidence, frameworks) EV->>EV: Index evidence by type (O(1) lookup) loop For each framework × control EV->>EV: Domain dispatch → check function end EV-->>O: FindingRecord[] + scores O->>RP: generate_reports(results, config) RP-->>O: HTML, JSON, MD, Excel, OSCAL, SARIF, PDF O-->>U: AssessmentSummary
Pipeline details:
  1. Discovery — discover_collectors() walks collectors/azure/ and collectors/entra/, importing modules which trigger @register_collector decorators
  2. Concurrent groups — Azure and Entra groups run in parallel. Within each, collectors execute in priority-sorted batches
  3. Error isolation — Each collector catches exceptions independently; failures produce access-denied markers
  4. Checkpoint resume — Interrupted assessments can resume from the last completed batch
  5. Severity-weighted scoring — critical=4, high=3, medium=2, low=1 with 50% partial credit

Safety Guarantees

🛡️

All Operations Are Strictly Read-Only

  • Azure SDK: exclusively list() and get()
  • Graph API: exclusively GET requests
  • No PUT, POST, PATCH, or DELETE — ever
  • No resource state modified, created, or deleted
  • No secret values read — only metadata
  • PII summarized (counts/rates), not exported
  • Deterministic UUID5 IDs for reproducibility
  • Checkpoint/resume for interrupted assessments
