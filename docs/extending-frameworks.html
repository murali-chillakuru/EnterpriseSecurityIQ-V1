Extension Pipeline

sequenceDiagram participant FM as Framework JSON participant E as Engine participant C as Collectors (68) participant EI as Evidence Index participant EV as Domain Evaluators participant P as Plugins FM->>E: Load controls array E->>C: discover_collectors() C->>EI: Produce evidence (keyed by EvidenceType) E->>EI: _index_evidence() → dict[str, list] loop For each control E->>EV: Primary dispatch by domain alt Handler returns not_assessed E->>EV: Cross-domain fallback (28 routes) end alt control has custom_evaluator E->>P: Plugin handles evaluation end end EV-->>E: Findings + status P-->>E: Findings + status
How extensions plug in:
  1. Framework JSON defines controls with domain, evidence_types, and evaluation_logic fields.
  2. Collectors produce evidence records keyed by EvidenceType. The engine builds an index for fast lookup.
  3. Evaluators dispatch to domain-specific handler functions. If the primary handler returns not_assessed, cross-domain fallback (28 routes) redirects to an alternative domain.
  4. Plugins handle controls with a custom_evaluator field, bypassing the domain dispatch entirely.

📋 Adding a New Framework

1 Create the Mapping File

Create AIAgent/app/frameworks/my-framework-mappings.json:

{
  "framework": "MY-FRAMEWORK",
  "frameworkName": "My Custom Framework",
  "version": "1.0",
  "description": "Custom compliance framework for internal use",
  "controls": [
    {
      "control_id": "MF-1.1",
      "title": "Access Control Policy",
      "domain": "access",
      "severity": "high",
      "evidence_types": ["azure-role-assignment", "entra-directory-role"],
      "evaluation_logic": "check_privileged_access_separation",
      "rationale": "Organization shall establish access control policies",
      "recommendation": "Review and restrict privileged role assignments"
    },
    {
      "control_id": "MF-2.1",
      "title": "Encryption at Rest",
      "domain": "data_protection",
      "severity": "high",
      "evidence_types": ["azure-keyvault", "azure-storage-security"],
      "evaluation_logic": "check_encryption_at_rest",
      "compensating_evidence": ["azure-disk-encryption-set"]
    }
  ]
}

2 Register the Framework

Add to AVAILABLE_FRAMEWORKS in AIAgent/app/evaluators/engine.py:

AVAILABLE_FRAMEWORKS = {
    "FedRAMP":       "fedramp-mappings.json",
    # ... existing ...
    "MY-FRAMEWORK":  "my-framework-mappings.json",
}

3 Use It

python run_assessment.py --tenant "..." --framework MY-FRAMEWORK

📦 Evidence Types

The engine matches controls to evidence using an index built from collector output. Each evidence record has an EvidenceType field. The engine's _index_evidence() builds a dict[str, list] keyed by this field.

📊 Evidence Matching

  2. Fallback: If primary yields nothing, compensating_evidence keys are checked

📡 Adding a New Collector

1 Create the File

# AIAgent/app/collectors/azure/my_source.py
from app.collectors.registry import register_collector

@register_collector(
    name="my_source",
    plane="control",      # "control" = ARM, "data" = data-plane
    source="azure",       # "azure" or "entra"
    priority=150          # lower = runs earlier
)
async def collect_azure_my_source(creds, subscriptions):
    evidence = []
    for sub in subscriptions:
        evidence.append({
            "EvidenceType": "azure-my-source",
            "SubscriptionId": sub["subscriptionId"],
            "ResourceId": "/subscriptions/.../my-resource",
            "Data": { "propertyA": "value" }
        })
    return evidence
✅ Auto-Discovered No manual import or registration needed. The @register_collector decorator registers the function. discover_collectors() uses pkgutil.iter_modules() to find and import all collector modules at startup.

⚙️ Adding a New Evaluator Function

1 Add the Handler

# In evaluators/access.py (or appropriate domain file)
def _check_my_custom_logic(control_id, control, evidence, evidence_index, thresholds):
    findings = []
    relevant = evidence_index.get("azure-my-source", [])
    for record in relevant:
        data = record.get("Data", {})
        if data.get("propertyA") != "expected":
            findings.append({
                "ControlId": control_id,
                "Status": "non_compliant",
                "Severity": control.get("severity", "medium"),
                "ResourceId": record.get("ResourceId", "N/A"),
                "Description": f"Property A is incorrect on {record.get('ResourceId')}",
                "Recommendation": control.get("recommendation", "Fix property A"),
            })
    if not relevant:
        findings.append({
            "ControlId": control_id,
            "Status": "missing_evidence",
            "Severity": control.get("severity", "medium"),
            "Description": "No azure-my-source evidence collected",
        })
    return findings

🔀 Evaluation Dispatch & Fallback

Two-Stage Dispatch with Cross-Domain Fallback

sequenceDiagram participant E as Engine participant PM as Primary Domain participant FM as _CROSS_DOMAIN_MAP participant AM as Alternative Domain E->>E: Read control.domain → "governance" E->>PM: governance.evaluate_governance(control) alt Handler returns findings PM-->>E: Findings ✅ else All findings are not_assessed PM-->>E: not_assessed E->>FM: Lookup evaluation_logic in _CROSS_DOMAIN_MAP FM-->>E: Alternative domain = "access" E->>AM: access.evaluate_access(control) AM-->>E: Findings ✅ (or missing_evidence) end
Two-stage dispatch:
  1. Primary — The control's domain field selects the domain evaluator via DOMAIN_EVALUATORS dict in engine.py.
  2. Fallback — If the primary evaluator returns not_assessed for all findings, the engine looks up evaluation_logic in _CROSS_DOMAIN_MAP (28 routes) and dispatches to an alternative domain.

This ensures controls get evaluated even when the primary domain lacks the evidence — e.g., a governance control that requires access data falls back to the access evaluator.

🔌 Plugin System

Creating a Plugin

# plugins/my_custom_check.py
def evaluate(control: dict, evidence_index: dict[str, list]) -> tuple[str, list[dict]]:
    relevant = evidence_index.get("azure-my-source", [])
    if not relevant:
        return "missing_evidence", [{
            "ControlId": control["control_id"],
            "Status": "missing_evidence",
            "Description": "No evidence found",
        }]
    findings = []
    for record in relevant:
        findings.append({
            "ControlId": control["control_id"],
            "Status": "compliant",
            "ResourceId": record.get("ResourceId", "N/A"),
        })
    return "compliant", findings

📐 The Formal JSON Schema

Defined in schemas/compliance-control.schema.json:

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "EnterpriseSecurityIQ Compliance Control Mapping",
  "required": ["framework", "frameworkName", "controls"],
  "properties": {
    "framework":    { "type": "string" },
    "frameworkName": { "type": "string" },
    "version":      { "type": "string" },
    "controls": {
      "type": "array",
      "items": {
        "required": ["control_id", "title", "domain",
                     "severity", "evidence_types", "evaluation_logic"],
        "properties": {
          "control_id":       { "type": "string" },
          "title":            { "type": "string" },
          "domain":           { "type": "string" },
          "severity":         { "enum": ["critical","high","medium","low","informational"] },
          "evidence_types":   { "type": "array", "items": { "type": "string" } },
          "evaluation_logic": { "type": "string" }
        }
      }
    }
  }
}
⚠️ Schema vs. Practice The schema's domain enum lists only 6 values. The engine supports all 10. The schema also omits compensating_evidence and custom_evaluator. Use the control fields table above as the authoritative reference.
