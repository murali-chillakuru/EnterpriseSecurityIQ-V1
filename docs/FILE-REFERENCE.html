Pipeline Overview

The assessment pipeline runs in four phases — authenticate, collect, evaluate, and report. Each phase is modular and can be invoked independently via CLI scripts or agent tools.

Phase 1
Authenticate
4 auth modes via ComplianceCredentials: auto, azurecli, serviceprincipal, appregistration. Unified DefaultAzureCredential for ARM + Graph. Preflight permissions check.
Phase 2
Collect
68 async collectors in concurrent batches. 47 registered Azure + 17 Entra + 2 standalone + M365. Checkpoint/resume. 200+ evidence types.
Phase 3
Evaluate
10 domain evaluators (113 functions). Severity-weighted scoring. Maps evidence → 525 controls across 11 frameworks. Suppression rules + plugin hooks.
Phase 4
Report
24 generators → HTML, MD, Excel, PDF, JSON, CSV, OSCAL, SARIF. Delta/drift/trending. Remediation playbooks. Per-framework sub-folders.

End-to-End Pipeline Sequence

sequenceDiagram
    participant CLI as CLI / Agent Tool
    participant Auth as auth.py
    participant Orch as postureiq_orchestrator.py
    participant Reg as registry.py
    participant AzC as Azure Collectors (47)
    participant EnC as Entra Collectors (17)
    participant Eval as Evaluators (10)
    participant Rep as Reports (24)

    CLI->>Auth: preflight_check()
    Auth-->>CLI: permissions verified

    CLI->>Orch: run_full_assessment(creds, config)
    Orch->>Reg: discover_collectors()
    Reg-->>Orch: sorted by priority

    par Azure + Entra concurrent
        Orch->>AzC: batch by azure_batch_size (6)
        AzC-->>Orch: evidence dicts
    and
        Orch->>EnC: batch by entra_batch_size (4)
        EnC-->>Orch: evidence dicts
    end

    Note over Orch: Checkpoint saved after each batch

    loop Each framework (11)
        Orch->>Eval: evaluate(evidence, controls)
        Eval-->>Orch: findings + control results
        Orch->>Rep: generate reports (HTML/MD/Excel/...)
        Rep-->>Orch: files written to sub-folder
    end

    Orch-->>CLI: AssessmentSummary
How it works: The orchestrator auto-discovers collectors via the @register_collector decorator, sorts them by priority, and dispatches them in concurrent batches. Azure and Entra groups run in parallel. After all evidence is collected, the evaluation engine iterates each framework's 525 controls, dispatching to the appropriate domain evaluator. Finally, 24 report generators produce per-framework outputs in timestamped sub-folders.

Evaluation Dispatch Flow

sequenceDiagram
    participant Orch as orchestrator.py
    participant FW as Framework JSON (525 controls)
    participant Eng as engine.py
    participant Dom as Domain Evaluator
    participant Sup as suppressions.py
    participant Plug as plugins.py

    Orch->>FW: Load controls for framework
    loop Each control
        Orch->>Eng: evaluate_control(evidence, control)
        Eng->>Eng: Match evaluation_logic → domain
        alt Known domain
            Eng->>Dom: domain.evaluate(evidence, control)
            Dom-->>Eng: finding + result
        else Unknown domain
            Eng->>Eng: Cross-domain fallback
            Eng-->>Eng: finding based on evidence presence
        end
        Eng->>Plug: run_plugins(finding)
        Plug-->>Eng: modified finding
        Eng->>Sup: check_suppression(finding)
        Sup-->>Eng: suppress / downgrade / pass
        Eng-->>Orch: final finding + weighted score
    end
    Orch->>Orch: Aggregate compliance % per domain
How evaluation works: For each framework control, the engine reads the evaluation_logic field to determine which domain evaluator handles it. The evaluator examines the collected evidence and returns a finding (compliant / non-compliant / partial / missing_evidence). The suppression engine can then downgrade or suppress findings based on configured rules. The engine calculates a severity-weighted compliance score: Critical(4), High(3), Medium(2), Low(1) weights.

Agent Tool Invocation Flow

sequenceDiagram
    participant User as User / LLM
    participant Agent as agent.py (14 tools)
    participant State as _session_state (Lock)
    participant Engine as Backing Engine

    User->>Agent: Tool call (e.g., assess_data_security)
    Agent->>State: acquire asyncio.Lock
    State-->>Agent: lock acquired
    Agent->>Engine: run_data_security_assessment(creds)
    Engine-->>Agent: results dict
    Agent->>State: store results in session
    State-->>Agent: stored
    Agent-->>User: formatted response

    Note over User,Agent: Subsequent tools (query_results, compare_runs) read from session state
Session state flow: Each tool stores its results in a thread-safe _session_state dictionary. Downstream tools like query_results and compare_runs read from this state, enabling multi-turn conversations where the agent remembers previous assessment results.

Safety Guarantees

  • Read-only access — all collectors use read-only Azure SDK and Graph API calls; no write operations
  • No PII in output — user_details collects aggregate lifecycle metrics, never names/emails/UPNs
  • Credential isolation — credentials never logged, serialized, or stored in evidence
  • Deterministic evaluation — 5 determinism test suites validate identical output for identical input
  • Graceful degradation — 403/401 errors produce AccessDenied evidence instead of failures
  • Configurable timeouts — per-collector timeout prevents runaway API calls
  • Checkpoint recovery — resumes from last successful batch on restart
  • Thread-safe state — asyncio.Lock protects shared session state in agent mode
