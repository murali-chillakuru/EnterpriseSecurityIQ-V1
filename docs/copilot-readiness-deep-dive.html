Overview

The M365 Copilot Readiness Engine evaluates whether an organization is prepared to safely deploy Microsoft 365 Copilot. Unlike a traditional compliance assessment that scores how well controls are implemented, this engine uses an inverted scoring model — starting at 100 (fully ready) and subtracting risk points for each gap found. The result is a single readiness score reflecting how much risk remains before Copilot can be enabled safely.

The engine executes ~85 checks across 10 categories spanning oversharing risk, sensitivity label coverage, DLP enforcement, data access governance, content lifecycle, audit readiness, Copilot-specific security, zero trust alignment, shadow AI controls, and restricted SharePoint search. Each finding is tagged with an effort level (quick_win, moderate, or major) so remediation teams can prioritize work.

Under the hood, the engine collects evidence from 15+ Microsoft Graph API calls, SharePoint/OneDrive inventory, sensitivity label configuration, DLP rules, conditional access policies, Entra ID roles, and more. It builds 9 inventory snapshots for auditability and constructs a 65-control security controls matrix mapping each control to PASS/FAIL with Microsoft Learn references.

Assessment Pipeline

The Copilot Readiness assessment follows a sequential pipeline: CLI entry → engine initialization → parallel evidence collection from SharePoint, Labels, Conditional Access, and Graph → 10 category analyzers (~85 checks) → 65-control security matrix → 9 inventory snapshots → scoring → multi-format report output.

Copilot Readiness Assessment Sequence
sequenceDiagram participant U as User / CI Pipeline participant R as run_copilot_readiness.py participant E as CopilotReadinessEngine participant SPO as SharePoint Collector participant LBL as Label Collector participant CA as CA Policy Collector participant G as Microsoft Graph participant S as Scoring Engine participant RP as Report Generator U->>R: python run_copilot_readiness.py --tenant T R->>E: run_copilot_readiness_assessment(creds) E->>SPO: collect_sharepoint_onedrive() SPO-->>E: Site inventory, permissions, sharing E->>LBL: collect_m365_sensitivity_labels() LBL-->>E: Labels, policies, DLP rules E->>CA: collect_entra_conditional_access() CA-->>E: CA policies, users, roles E->>G: 15+ direct Graph API calls G-->>E: SKUs, access reviews, PIM, apps E->>E: Run 10 category analyzers (~85 checks) E->>E: build_security_controls_matrix() — 65 controls E->>E: Build 9 inventory snapshots E->>S: compute_copilot_readiness_scores() S-->>E: Readiness scores + status E-->>R: Full assessment result R->>RP: HTML + CSV + JSON + PDF RP-->>U: Output files
  1. CLI Entry — run_copilot_readiness.py parses arguments, authenticates, and hands off to the engine.
  2. Evidence Collection — SharePoint sites, sensitivity labels, CA policies, and 15+ Graph API calls gather tenant state.
  3. Category Analyzers — 10 analyzers run ~85 checks, each producing findings with severity and effort tags.
  4. Security Controls Matrix — 65 controls are evaluated to PASS/FAIL with Microsoft Learn references.
  5. Scoring — Inverted model: Readiness = 100 − risk points. Status determined by thresholds.
  6. Reporting — HTML dashboard, CSV details, JSON machine-readable, optional PDF.

Oversharing Risk

The oversharing_risk category is the most critical assessment area — Copilot inherits all user permissions, so any overshared content becomes instantly searchable and surfaceable. This category evaluates 12 checks covering site discovery, broad membership, anonymous links, external sharing, and SharePoint Advanced Management (SAM) features.

CheckSeverityWhat It Evaluates
unable_to_assessHighSharePoint data collection failed entirely — no site inventory available for analysis
partial_site_discoveryMediumOnly a subset of SharePoint sites were discovered — permissions audit is incomplete
broad_site_membershipHighSites with large membership groups (e.g., "Everyone except external users") granting broad access
everyone_permissionsCritical"Everyone" or "Everyone except external users" has explicit permissions on sites or libraries
anonymous_link_exposureCriticalAnonymous sharing links exist that allow unauthenticated access to content Copilot can surface
external_sharing_postureHighOrg-wide external sharing settings are overly permissive (e.g., "Anyone" level enabled)
no_sharepoint_advanced_managementMediumSharePoint Advanced Management (SAM) license not detected — advanced governance features unavailable
no_sam_restricted_access_controlMediumSAM Restricted Access Control policies not configured to limit site-level access
no_sam_site_lifecycle_policyMediumNo SAM site lifecycle policies to automatically manage inactive or stale sites
no_sam_dag_reportsMediumSAM Data Access Governance reports not enabled for oversharing visibility
high_permission_blast_radiusCriticalCalculated blast radius of broad permissions — percentage of content accessible to most users
external_sharing_risk_scoreHighComposite risk score based on sharing link types, guest counts, and external domain spread
Why this matters: Microsoft 365 Copilot respects existing permissions — it never bypasses access controls. But if "Everyone except external users" has read access to a finance site, Copilot will happily summarize those documents for any employee who asks. Fixing oversharing is the #1 prerequisite for safe Copilot deployment.

Security Controls Matrix

The engine builds a 65-control security controls matrix via build_security_controls_matrix(). Each control maps to a specific Microsoft security capability, is evaluated to PASS or FAIL, and includes a direct link to the relevant Microsoft Learn documentation.

How it works: The 65 controls span all 10 assessment categories. For each control, the engine checks whether the corresponding configuration, policy, or feature is present and properly configured in the tenant. Controls are not weighted individually — they contribute to the category-level scoring which feeds the overall readiness score. The matrix provides a single-page audit artifact showing the complete security posture for Copilot deployment.
FieldDescriptionExample
control_idUnique identifier for the controlCR-OVR-001
categoryAssessment category the control belongs toOversharing Risk
control_nameHuman-readable nameSharePoint External Sharing Restricted
descriptionWhat the control verifiesExternal sharing limited to authenticated guests or more restrictive
statusEvaluation resultFAIL / PASS
learn_urlMicrosoft Learn referencehttps://learn.microsoft.com/sharepoint/...
remediationRecommended fix actionRestrict sharing to "Existing guests" or "Only people in your org"
Audit-ready output: The controls matrix is exported as a dedicated CSV file alongside the main report — ideal for sharing with security teams, auditors, or compliance reviewers who need a structured checklist of Copilot security controls.

Scoring Model

The Copilot Readiness Engine uses an inverted readiness model — fundamentally different from the compliance assessment. Instead of scoring how many controls pass, it starts at 100 and subtracts risk points for each gap found. Readiness = 100 − total_risk_points.

Inverted scoring rationale: A compliance score rewards what you've done. A readiness score penalizes what you haven't. This inversion ensures organizations focus on the gaps that prevent safe Copilot deployment rather than celebrating existing controls that may be irrelevant to Copilot risk.

Readiness Levels

Score RangeStatusMeaning
≥ 75ReadyOrganization has addressed critical risks and can proceed with Copilot deployment
≥ 50Mostly ReadySome gaps remain but can be addressed during phased rollout
≥ 25Needs WorkSignificant gaps requiring remediation before Copilot enablement
< 25Not ReadyCritical security gaps — Copilot deployment would expose sensitive data

Status Determination Rules

The engine applies additional rules beyond the raw score: any Critical severity finding automatically caps the status at "Needs Work" regardless of the numerical score. Three or more High severity findings cap the status at "Mostly Ready". This ensures critical risks cannot be hidden by a high count of passing low-severity checks.

Effort Tags

TagDescriptionTypical Timeline
quick_winConfiguration change, no new licensing or infrastructure needed1–3 days
moderateRequires policy design, testing, and staged rollout1–4 weeks
majorRequires new licensing, infrastructure, or organizational change1–3 months

Each of the ~80 effort mappings associates a specific finding with an effort tag, enabling remediation teams to sort by effort and tackle quick wins first before committing to larger projects.

CLI Usage

run_copilot_readiness.py is the standalone CLI entry point. It authenticates, invokes the engine, and produces multi-format output.

# Basic usage with Azure CLI auth
python run_copilot_readiness.py --tenant contoso.onmicrosoft.com

# Service principal authentication
python run_copilot_readiness.py \
    --tenant contoso.onmicrosoft.com \
    --auth-mode serviceprincipal \
    --client-id <app-id> \
    --client-secret <secret>

# Custom output directory
python run_copilot_readiness.py --tenant contoso.onmicrosoft.com --output ./reports

# Skip PDF generation
python run_copilot_readiness.py --tenant contoso.onmicrosoft.com --no-pdf

# JSON-only output for CI/CD integration
python run_copilot_readiness.py --tenant contoso.onmicrosoft.com --format json
ArgumentRequiredDescription
--tenantYesAzure AD tenant ID or domain name
--auth-modeNoAuthentication mode: auto (default), azurecli, serviceprincipal, appregistration
--client-idNoApp registration client ID (required for serviceprincipal/appregistration)
--client-secretNoApp registration client secret
--outputNoOutput directory path (default: ./output/<timestamp>)
--formatNoOutput format: all (default), html, json, csv
--no-pdfNoSkip PDF report generation
--configNoPath to custom configuration file
--verboseNoEnable detailed logging output
