Assessment Pipeline

The orchestrator coordinates a 3-phase pipeline: collect → evaluate → report. Azure and Entra collectors run concurrently in batches.

Evidence Collection Flow
sequenceDiagram participant U as User / Agent participant O as postureiq_orchestrator.py participant R as Registry participant AZ as Azure Collectors (49) participant EN as Entra Collectors (17) participant EV as Evidence Store U->>O: run_full_assessment(creds, config) O->>R: discover_collectors() R-->>O: 49 Azure + 17 Entra functions par Azure Collection (batch_size=8) O->>AZ: Batch 1: resources, rbac, policy, diagnostics... AZ-->>O: evidence[] + timings O->>AZ: Batch 2: activity_logs, security, network... AZ-->>O: evidence[] + timings O->>AZ: Batch N: remaining collectors AZ-->>O: evidence[] + timings and Entra Collection (batch_size=6) O->>EN: Batch 1: tenant, users, user_details, CA... EN-->>O: evidence[] + timings O->>EN: Batch 2: roles, apps, governance... EN-->>O: evidence[] + timings end O->>EV: Merge all evidence (200+ types) Note over O,EV: Checkpoint saved after collection
How collection works:
  1. Discovery — discover_collectors() walks collectors/azure/ and collectors/entra/, importing each module which triggers @register_collector decorators
  2. Concurrent groups — Azure and Entra collector groups run in parallel via asyncio.gather()
  3. Batched execution — Within each group, collectors run in batches (default: 8 Azure, 6 Entra) to avoid API throttling
  4. Error isolation — Each collector catches exceptions independently; failures produce access-denied markers, not pipeline failures
  5. Checkpoint resume — After each batch, a checkpoint file is saved so interrupted assessments can resume
Evaluation Pipeline
sequenceDiagram participant EV as Evidence Store participant ENG as postureiq_evaluators/engine.py participant FW as Framework Mappings (11) participant DOM as Domain Evaluators (10) participant SC as Scoring Engine EV->>ENG: evaluate_all(evidence, frameworks) ENG->>ENG: _index_evidence() → O(1) lookup by type loop For each framework ENG->>FW: Load JSON mapping (controls + domains) FW-->>ENG: Control definitions with evidence_types loop For each control ENG->>DOM: Dispatch to domain evaluator DOM->>DOM: Check evidence against thresholds alt Primary evidence missing DOM->>ENG: Try compensating_evidence end alt Status = not_assessed ENG->>DOM: Cross-domain fallback end DOM-->>ENG: FindingRecord[] + status end end ENG->>SC: Severity-weighted scoring Note over SC: critical=4, high=3, medium=2, low=1 SC-->>ENG: Per-domain scores + overall score
How evaluation works:
  1. Evidence indexing — All evidence records are indexed by EvidenceType into a dict for O(1) lookup
  2. Framework loading — 11 JSON mapping files define controls with domain, evidence_types, handler, and compensating_evidence
  3. Domain dispatch — DOMAIN_EVALUATORS maps each domain to its evaluator module (10 domains → 113 check functions)
  4. Cross-domain fallback — If primary evaluator returns not_assessed, _CROSS_DOMAIN_MAP re-dispatches (e.g., storage_security → network)
  5. Scoring — Severity-weighted with partial credit (50%); missing_evidence excluded from compliance percentage
Agent Tool Invocation Flow
sequenceDiagram participant User as User participant F as Microsoft Foundry participant A as agent.py (TOOLS) participant CR as ComplianceCredentials participant API as Azure ARM / MS Graph User->>F: Natural language request F->>A: Select and call tool function A->>CR: ComplianceCredentials() CR->>CR: DefaultAzureCredential CR-->>A: ARM client + Graph client alt run_postureiq_assessment A->>API: 68 collectors (ARM + Graph) API-->>A: Raw evidence A->>A: evaluate_all() → findings A->>A: Generate reports A-->>F: Markdown summary else search_tenant A->>API: ARG KQL or Graph query API-->>A: Results A-->>F: Formatted results else analyze_risk A->>API: Targeted evidence collection API-->>A: Risk indicators A->>A: Severity scoring A-->>F: Risk analysis report end F-->>User: Formatted response
How agent tools work:
  1. Tool registration — Each tool is an async function with Annotated type hints, registered in the TOOLS list
  2. Authentication — ComplianceCredentials creates a unified credential for both ARM and Graph operations
  3. Session state — Assessment results are cached in _session_state (thread-safe with asyncio.Lock)
  4. Evidence reuse — Risk analysis and data security tools reuse cached evidence when available

Standalone Collectors (Not Auto-Registered)

2 collectors that are intentionally NOT part of the auto-discovery pipeline. They are called directly by specific engines.

S1 RBAC Collector — rbac_orchestrator.py

Called directly by run_rbac_report.py and the generate_rbac_report agent tool. Uses rbac_evaluators/ for modular evaluation. Returns a hierarchical tree (not flat evidence).

6-Phase Process:

  1. Build management group tree
  2. MG-level assignments + PIM eligibility
  3. Per-subscription assignments + RGs + resources
  4. Attach subscriptions to MG tree
  5. Resolve principal display names via Graph getByIds (batch 1000)
  6. Expand group memberships (transitive members)
S2 AI Identity Collector — ai_identity.py (419 lines)

Called directly by the AI Agent Security engine. Identifies AI-related service principals using Microsoft AI app IDs + display name heuristics.

Evidence: entra-ai-service-principal entra-ai-consent-grant entra-cross-tenant-policy
