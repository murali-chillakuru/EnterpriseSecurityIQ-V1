Overview

Dual Delivery Model

PostureIQ uses a combined Option A+B architecture. A single Container App serves both a SPA web dashboard (Option A) and registers as a Foundry Agent (Option B) via the Assistants API on startup. Both paths share the same 14 agent tools and Azure OpenAI backend.

🌐 Option A — SPA Dashboard

Web dashboard served at / with MSAL.js SSO authentication. Provides assessment runner, results viewer, and agent chat interface.

GET / → Static HTML/JS/CSS

🤖 Option B — Foundry Agent

Agent registered in Foundry project via Assistants API on container startup. Visible in ai.azure.com with tracing, evaluations, and guardrails.

Agent ID: asst_N4hpInCl30eZHaim3vtJTZiT

Authentication Flows

Option A — SPA Authentication

User → Browser → MSAL.js (Entra ID SSO)
  → Acquires Bearer token
  → POST /chat with Authorization: Bearer <token>
  → FastAPI validates token
  → Calls Azure OpenAI with function calling
  → Agent tools execute via Managed Identity

Option B — Foundry Agent Authentication

User → ai.azure.com portal → Foundry Agent (Assistants API)
  → Azure OpenAI processes with function calling
  → Agent tools execute via Managed Identity

Foundry Agent Registration

On container startup, the agent self-registers in the Foundry project via the Assistants API:

  1. Container App starts FastAPI server on port 8088
  2. Startup handler calls Azure OpenAI Assistants API
  3. Creates/updates assistant with name EnterpriseSecurityIQ
  4. Registers all 14 tools as function definitions
  5. Agent becomes visible in ai.azure.com under the Foundry project
  6. Agent ID: asst_N4hpInCl30eZHaim3vtJTZiT

Foundry Portal Features

📊 Tracing

Built-in request/response tracing via Application Insights integration.

📈 Evaluations

Run evals against the agent directly from the ai.azure.com portal.

🛡️ Guardrails

Content safety filters applied at the Azure OpenAI layer.
