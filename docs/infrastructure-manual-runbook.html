1. Understanding the Architecture

PostureIQ is an AI-powered compliance assessment platform that runs as a Docker container on Azure Container Apps. It uses Azure AI Foundry for LLM reasoning, MSAL.js for browser-based authentication, and user-delegated tokens so assessments run under the user's own Entra permissions.

Key Principle: Assessments run under the user's own identity. The managed identity is only for infrastructure (pulling Docker images, calling the LLM).
End-to-End Deployment Flow
sequenceDiagram participant Op as Operator participant CLI as Azure CLI participant ARM as Azure ARM participant Entra as Entra ID participant ACR as Container Registry participant CA as Container App Note over Op,CA: Phase 1 — Core Infrastructure (Steps 0–10) Op->>CLI: az login + set variables CLI->>ARM: Create RG, Foundry, Models, Storage, KV, Monitoring, ACR Note over Op,CA: Phase 2 — Identity & RBAC (Steps 11–12) CLI->>ARM: Create Managed Identity CLI->>ARM: Assign 5 Azure RBAC roles CLI->>Entra: Grant 8 Graph application permissions Note over Op,CA: Phase 3 — Build & Deploy (Steps 13–16) CLI->>ACR: Build Docker image (ACR Tasks) CLI->>CA: Create Container Apps Environment + App CLI->>Entra: Create App Registration (SPA + PKCE) CLI->>ACR: Patch SPA config → rebuild → update Note over Op,CA: Post-Deployment (Manual) Op->>Entra: Grant admin consent (portal) Op->>Entra: Assign Directory Reader + Global Reader (portal) CA-->>Op: Dashboard live at FQDN
  1. Phase 1 (Steps 0–10): Log in, define variables, create all core Azure infrastructure — resource group, AI Foundry (custom domain + project management), model deployments, storage, key vault, monitoring stack, and container registry.
  2. Phase 2 (Steps 11–12): Create the managed identity, assign 5 Azure RBAC roles and 8 Microsoft Graph application permissions.
  3. Phase 3 (Steps 13–16): Create the Container Apps Environment, build/deploy the Docker image, register the SPA in Entra ID, patch MSAL config and redeploy.
  4. Post-Deployment: Two manual Entra portal steps require Global Administrator — grant admin consent and assign directory roles.

Steps 1–3 — RG, Foundry Resource & Project

Step 1 — Resource Group

A Resource Group is a logical container for related Azure resources. All 13 Azure resources live here. Deleting the RG deletes everything inside it.

$rgExists = az group exists --name $RG -o tsv
if ($rgExists -eq "false") {
    az group create --name $RG --location $Location -o table
    Write-Host "Created resource group: $RG"
} else {
    Write-Host "Resource group $RG already exists — skipping."
}

Step 2 — Foundry Resource (AI Services)

Azure AI Foundry is Microsoft's platform for building AI applications. Under the hood, it's a CognitiveServices/accounts resource with kind AIServices. Two special configurations are needed: a custom domain (required before projects can be created) and allowProjectManagement (enables the "New Foundry" project model).

# CREATE THE FOUNDRY RESOURCE
$aiExists = az cognitiveservices account show --name $AIName --resource-group $RG `
    --query "name" -o tsv 2>$null

if (-not $aiExists) {
    az cognitiveservices account create `
        --name $AIName --resource-group $RG `
        --kind "AIServices" --sku "S0" --location $Location --yes -o none
}

# SET CUSTOM DOMAIN (required for Foundry projects)
az cognitiveservices account update `
    --name $AIName --resource-group $RG --custom-domain $CustomDomain -o none

# ENABLE PROJECT MANAGEMENT (ARM REST API)
$aiUri = "https://management.azure.com/subscriptions/$SubId/resourceGroups/$RG/providers/Microsoft.CognitiveServices/accounts/${AIName}?api-version=2025-04-01-preview"
$bodyFile = [System.IO.Path]::GetTempPath() + "esiq-allow-pm.json"
'{"properties":{"allowProjectManagement":true}}' | Set-Content $bodyFile -Encoding UTF8
az rest --method PATCH --uri $aiUri --body "@$bodyFile" -o none
Remove-Item -Force $bodyFile -ErrorAction SilentlyContinue

# CAPTURE OUTPUT VARIABLES
$AIEndpoint = az cognitiveservices account show --name $AIName --resource-group $RG `
    --query "properties.endpoint" -o tsv
$AIId = az cognitiveservices account show --name $AIName --resource-group $RG `
    --query "id" -o tsv
Write-Host "AI Endpoint: $AIEndpoint"
What is az rest? — The Azure CLI's tool for calling any Azure REST API directly. We use it here because allowProjectManagement is a new property not yet exposed as a CLI parameter.
✓ Verify: 
az cognitiveservices account show --name $AIName --resource-group $RG `
    --query "{name:name, kind:kind, endpoint:properties.endpoint, customDomain:properties.customSubDomainName}" -o table
Learn More: Azure AI Foundry — What is it? CognitiveServices REST API az rest — Call any Azure REST API

Steps 4–5 — Model Deployments

Model Request Flow
sequenceDiagram participant App as Container App participant P as gpt-4.1 (Primary) participant F as gpt-5.1 (Fallback) App->>P: Tool call / reasoning request alt Success P-->>App: Response (tokens) else Throttled / Complex task App->>F: Retry with fallback model F-->>App: Response (tokens) end
  1. Primary Model (gpt-4.1): Handles most operations — tool calls, reasoning, report generation.
  2. Fallback (gpt-5.1): Used for complex tasks or when the primary model is throttled. The application code handles failover automatically.

Step 4 — Primary Model: gpt-4.1

A Model Deployment makes an LLM available for API calls. The deployment name becomes the identifier used when calling the API.

$PrimaryModel  = "gpt-4.1"
$ModelSku      = "Standard"
$ModelCapacity = 30   # 30K TPM (tokens per minute)

$m1Exists = az cognitiveservices account deployment show `
    --name $AIName --resource-group $RG `
    --deployment-name $PrimaryModel --query "name" -o tsv 2>$null

if (-not $m1Exists) {
    az cognitiveservices account deployment create `
        --name $AIName --resource-group $RG `
        --deployment-name $PrimaryModel `
        --model-name $PrimaryModel --model-version "2025-04-14" `
        --model-format "OpenAI" `
        --sku-name $ModelSku --sku-capacity $ModelCapacity -o none
}

Step 5 — Fallback Model: gpt-5.1

$FallbackModel = "gpt-5.1"

$m2Exists = az cognitiveservices account deployment show `
    --name $AIName --resource-group $RG `
    --deployment-name $FallbackModel --query "name" -o tsv 2>$null

if (-not $m2Exists) {
    az cognitiveservices account deployment create `
        --name $AIName --resource-group $RG `
        --deployment-name $FallbackModel `
        --model-name $FallbackModel --model-version "2025-11-13" `
        --model-format "OpenAI" `
        --sku-name $ModelSku --sku-capacity $ModelCapacity -o none

A Managed Identity is an Azure-managed service account — no passwords to manage. We use user-assigned (rather than system-assigned) because if the Container App is deleted and recreated, the identity and all its RBAC roles survive.

$PrincipalId = az identity show --name $IDName --resource-group $RG --query "principalId" -o tsv
$ClientId    = az identity show --name $IDName --resource-group $RG --query "clientId" -o tsv
$IdentityId  = az identity show --name $IDName --resource-group $RG --query "id" -o tsv

Step 12b — Microsoft Graph API Permissions (8)

Requires Global Administrator. This step grants application-level Graph permissions. If you don't have Global Admin, ask one to run these commands. The web dashboard still works without them — only CLI scripts are affected. 
# LOOK UP MICROSOFT GRAPH SERVICE PRINCIPAL
$GraphAppId = "00000003-0000-0000-c000-000000000000"
$graphSpResp = az rest --method GET `
    --uri "https://graph.microsoft.com/v1.0/servicePrincipals?`$filter=appId eq '$GraphAppId'&`$select=id,appRoles" 2>&1
$graphSpParsed = ($graphSpResp | Out-String | ConvertFrom-Json).value[0]
$GraphSpId     = $graphSpParsed.id
$graphRoles    = $graphSpParsed.appRoles

# GRANT 8 APPLICATION PERMISSIONS
$graphPerms = @(
    "Directory.Read.All", "Policy.Read.All", "RoleManagement.Read.All",
    "User.Read.All", "AuditLog.Read.All", "UserAuthenticationMethod.Read.All",
    "IdentityRiskyUser.Read.All", "Application.Read.All"
)

foreach ($permName in $graphPerms) {
    $role = $graphRoles | Where-Object { $_.value -eq $permName }
    if (-not $role) { Write-Host "  WARNING: $permName not found — skipping."; continue }

    $body = @{
        principalId = $PrincipalId
        resourceId  = $GraphSpId
        appRoleId   = $role.id
    } | ConvertTo-Json -Compress

    az rest --method POST `
        --uri "https://graph.microsoft.com/v1.0/servicePrincipals/$PrincipalId/appRoleAssignments" `
        --body $body --headers "Content-Type=application/json" -o none 2>$null
    Write-Host "  Granted: $permName"
}

User.Read.AllRead user profiles
AuditLog.Read.AllRead sign-in and audit logs
UserAuthenticationMethod.Read.AllRead MFA registration status
IdentityRiskyUser.Read.AllRead Identity Protection risky users
Application.Read.AllRead app registrations and service principals

Steps 15–16 — App Registration & SPA Patch

SPA Authentication Flow (PKCE)
sequenceDiagram participant User as User Browser participant SPA as SPA (MSAL.js) participant Entra as Entra ID participant CA as Container App API User->>SPA: Click "Login" SPA->>SPA: Generate PKCE code_verifier + code_challenge SPA->>Entra: GET /authorize (client_id, redirect_uri, code_challenge) Entra->>User: Login prompt User->>Entra: Username + password + MFA Entra-->>SPA: Authorization code (redirect) SPA->>Entra: POST /token (code + code_verifier) Entra-->>SPA: access_token (Graph) + id_token SPA->>Entra: acquireTokenSilent (ARM scope) Entra-->>SPA: access_token (ARM) SPA->>CA: POST /chat + Bearer tokens CA-->>SPA: SSE streamed response
  1. PKCE: SPA can't store client secrets, so MSAL.js generates a dynamic code_verifier/challenge pair for each login. This prevents authorization code interception attacks.
  2. Two Tokens: The SPA acquires separate access tokens for Microsoft Graph and Azure ARM — both are sent to the backend.
  3. Same-Origin: The SPA and API share the same Container App FQDN, so AGENT_URL is set to empty string (same-origin).

Step 15 — Entra App Registration (SPA)

An App Registration defines the application in Entra ID — its client ID, that it's a SPA using MSAL.js/PKCE, and its redirect URIs.

$existingApp = az ad app list --display-name $DashName --query "[0].appId" -o tsv 2>$null

if ($existingApp) {
    $AppClientId = $existingApp
} else {
    $appJson = az ad app create `
        --display-name $DashName `
        --sign-in-audience "AzureADMyOrg" `
        --enable-access-token-issuance true `
        --enable-id-token-issuance true -o json
    $AppClientId = ($appJson | ConvertFrom-Json).appId
}

# SET SPA REDIRECT URIs
az ad app update --id $AppClientId `
    --spa-redirect-uris "http://localhost:8080" "https://$FQDN" -o none

Write-Host "App Client ID: $AppClientId"

Step 16 — Patch SPA Config & Rebuild

The SPA has placeholder values for MSAL configuration. We replace them with real deployment values, rebuild the Docker image, and update the Container App.

$webappFile = Join-Path (Get-Location) "webapp" "index.html"

if (Test-Path $webappFile) {
    $html = Get-Content $webappFile -Raw -Encoding UTF8

    # Replace placeholders with live values
    $html = $html.Replace('clientId: "YOUR-CLIENT-ID-HERE"', "clientId: `"$AppClientId`"")
    $html = $html.Replace('YOUR-TENANT-ID-HERE', $TenantId)
    $html = $html.Replace('const AGENT_URL = "https://YOUR-AGENT-URL-HERE"', 'const AGENT_URL = ""')

    Set-Content $webappFile -Value $html -Encoding UTF8

    # REBUILD IMAGE WITH PATCHED SPA
    az acr build --registry $ACRName --image "${AppName}:v1" `
        --file "AIAgent/Dockerfile" "." --no-logs `
        --build-arg CACHEBUST=$(Get-Date -Format 'yyyyMMddHHmmss')

    # UPDATE CONTAINER APP
    az containerapp update --name $AppName --resource-group $RG `
        --image "${ACRName}.azurecr.io/${AppName}:v1" -o none
}
Why baked-in config? The SPA is a static HTML file needing clientId/tenantId at load time. Changing tenant or App Registration requires an image rebuild.
