Overview

The Risk Analysis Engine (risk_engine.py) is a 1,280-line attack-surface risk gap analysis engine that evaluates tenant security posture across five categories: Identity, Network, Defender, Configuration, and Insider Risk. It combines evidence-driven analysis from Azure Resource Graph and Microsoft Graph with live Defender API integration to produce a weighted risk score with configurable thresholds.

Design Philosophy: Attack-surface-first analysis with configurable thresholds. The engine collects lightweight evidence via Azure Resource Graph, enriches it with Microsoft Graph identity data and live Defender API calls, then runs 19 targeted checks across 5 risk categories. Every finding is backed by concrete evidence — resource IDs, configuration snapshots, and API responses.

Pipeline

The risk analysis pipeline collects evidence from three sources — Azure Resource Graph, Microsoft Graph, and the Defender API — then runs five category analyzers in sequence before computing weighted risk scores.

Risk Analysis Pipeline — Sequence Diagram
sequenceDiagram participant U as User / CI Pipeline participant R as run_risk_analysis.py participant E as RiskEngine participant ARG as Azure Resource Graph participant G as Microsoft Graph participant D as Defender API participant S as Scoring Engine U->>R: python run_risk_analysis.py --tenant T R->>E: run_risk_analysis(creds) E->>ARG: _lightweight_collect() ARG-->>E: Resource configs E->>G: Users, roles, apps, MFA G-->>E: Identity evidence E->>E: analyze_identity_risk() — 7 checks E->>E: analyze_network_risk() — 4 checks E->>D: analyze_defender_posture() — 3 checks D-->>E: Plans, recommendations, score E->>E: analyze_config_drift() — 3 checks E->>E: analyze_insider_risk() — 2 checks E->>S: compute_risk_scores() S-->>E: Scores + levels E-->>R: Full analysis R-->>U: Risk score + findings
  1. Entry point: run_risk_analysis.py parses CLI arguments and acquires credentials.
  2. Evidence collection: Lightweight resource graph queries fetch Azure resource configs; Microsoft Graph provides identity data (users, roles, apps, MFA status).
  3. Category analysis: Five analyzers run in sequence — Identity (7 checks), Network (4), Defender (3), Configuration (3), and Insider Risk (2).
  4. Defender API: Live calls to the Defender API retrieve plan coverage, security recommendations, and secure score.
  5. Scoring: Weighted scores are computed per category and aggregated into an overall risk score with severity level.
  6. Output: Full analysis results are returned with risk scores, findings, and evidence for report generation.

Identity Risk Checks

The identity risk analyzer evaluates 7 checks across user lifecycle, service principal permissions, credential hygiene, MFA enrollment, admin sprawl, guest access, and risky user signals from Entra ID Protection.

CheckSeverityThresholdWhat It Detects
dormant_accounts High >90 days, >20% User accounts with no sign-in activity for 90+ days; flags when stale accounts exceed 20% of total users
overpermissioned_sps Critical — Service principals with broad Microsoft Graph permissions (e.g., Directory.ReadWrite.All, Mail.ReadWrite) that violate least-privilege
app_credential_hygiene High / Medium — App registrations with expired or long-lived credentials (secrets >180 days, certificates approaching expiry)
mfa_gaps Critical <90% Users without MFA registration; critical when MFA coverage drops below 90% of enabled users
admin_proliferation High >5 GA Excessive Global Administrator role assignments; flags when count exceeds 5 permanent GA holders
guest_risks Medium >20% High guest-to-member ratio indicating potential over-sharing; flags when guests exceed 20% of directory users
risky_users Critical / High — Users flagged by Entra ID Protection with active risk detections (leaked credentials, anomalous travel, malware-linked sign-ins)

Network Risk Checks

The network risk analyzer evaluates 4 checks across management port exposure, public storage endpoints, web application security, and SQL database network exposure.

CheckSeverityThresholdWhat It Detects
open_management_ports Critical — NSG rules allowing inbound SSH (22) or RDP (3389) from Any/Internet source; immediate attack surface for brute force
public_storage High — Storage accounts with public network access enabled and no private endpoints; checks for blob anonymous access, public containers
webapp_security High / Medium — Web apps without HTTPS-only enforcement, missing managed identity, or using outdated TLS versions (<1.2)
sql_exposure Critical — Azure SQL servers with firewall rules allowing 0.0.0.0–255.255.255.255 or "Allow Azure Services" without private endpoints
🛡️

Defender Posture Checks

The Defender posture analyzer queries the Microsoft Defender for Cloud API to evaluate plan coverage, outstanding security recommendations, and the overall secure score.

CheckSeverityThresholdWhat It Detects
defender_coverage High — Defender plans not enabled for critical resource types (Servers, AppServices, SqlServers, Storage, KeyVaults, Containers)
security_recommendations Critical / High / Medium — Outstanding Defender recommendations by severity; counts unhealthy resources and maps to affected resource types
secure_score High / Medium <70% Microsoft Defender secure score below 70%; computes current-to-max ratio and identifies top improvement controls

Configuration & Insider Risk Checks

Configuration drift checks evaluate diagnostic logging coverage, Azure Policy compliance, and resource tag governance. Insider risk checks assess Insider Risk Management (IRM) policy existence and active alert signals.

Configuration Drift — 3 Checks

CheckSeverityThresholdWhat It Detects
diagnostic_coverage Medium — Resources without diagnostic settings configured; checks for Log Analytics workspace or Event Hub sink attachment
policy_noncompliance High / Medium — Azure Policy assignments in non-compliant state; counts non-compliant resources per policy definition
tag_governance Low — Resources missing required tags (Environment, Owner, CostCenter); measures tag coverage percentage

Insider Risk — 2 Checks

CheckSeverityThresholdWhat It Detects
irm_policy_existence Medium — Whether Insider Risk Management policies are configured in the tenant; absence indicates no insider threat monitoring
irm_active_alerts High — Active insider risk alerts that have not been triaged or resolved; counts open alerts by severity and policy type

Scoring Model

Each category produces a weighted score. The overall risk score is a weighted sum across all five categories, mapped to a severity level.

Category Weights

CategoryWeightRationale
Identity Risk30%Identity is the primary attack vector; compromised credentials enable lateral movement
Network Risk25%Exposed endpoints and open ports provide direct attack surface for external threats
Defender Posture20%Defender coverage and score reflect overall security monitoring and detection capability
Configuration Drift15%Misconfigurations and policy non-compliance create exploitable gaps
Insider Risk10%Insider threats are lower frequency but high impact; monitoring presence is the key signal
Formula: overall_score = (identity × 0.30) + (network × 0.25) + (defender × 0.20) + (config × 0.15) + (insider × 0.10)
Each category score ranges from 0–100, where 100 indicates maximum risk. The overall score is then mapped to a severity level.

Risk Levels

Score RangeLevelAction Required
≥ 75CriticalImmediate remediation required; executive escalation recommended
≥ 50HighPriority remediation within current sprint; security team review
≥ 25MediumScheduled remediation; include in next planning cycle
< 25LowAcceptable posture; continue monitoring

Configurable Thresholds

Three key thresholds can be customized in the configuration to tune sensitivity for your environment. These thresholds control when specific checks escalate to higher severity levels.

ThresholdDefaultConfig KeyDescription
max_stale_percent 20% risk_analysis.thresholds.max_stale_percent Maximum percentage of dormant accounts before the dormant_accounts check escalates to High severity
min_mfa_percent 90% risk_analysis.thresholds.min_mfa_percent Minimum MFA enrollment percentage; below this, the mfa_gaps check escalates to Critical severity
max_global_admins 5 risk_analysis.thresholds.max_global_admins Maximum number of permanent Global Administrator role assignments before admin_proliferation escalates to High
Tip: Override thresholds in your enterprisesecurityiq.config.json file under the risk_analysis.thresholds section. CI/CD pipelines can pass --config /path/to/config.json to use environment-specific thresholds.

CLI Usage

The risk analysis engine is invoked via the run_risk_analysis.py CLI entry point. It supports tenant targeting, output format selection, and configuration overrides. 

# Basic risk analysis for a tenant
python run_risk_analysis.py --tenant <TENANT_ID>

# With custom config and output directory
python run_risk_analysis.py --tenant <TENANT_ID> \
    --config config/enterprisesecurityiq.config.json \
    --output-dir ./output/risk-analysis

# Specify output format (json, html, markdown, csv)
python run_risk_analysis.py --tenant <TENANT_ID> --format html

# CI/CD mode with non-interactive auth
python run_risk_analysis.py --tenant <TENANT_ID> \
    --client-id <CLIENT_ID> \
    --client-secret <CLIENT_SECRET>

# Full example with all options
python run_risk_analysis.py \
    --tenant <TENANT_ID> \
    --config config/enterprisesecurityiq.config.json \
    --output-dir ./output/risk \
    --format json \
    --verbose
Exit Codes: 0 = success, 1 = critical findings detected (useful for CI/CD gate enforcement), 2 = execution error.
