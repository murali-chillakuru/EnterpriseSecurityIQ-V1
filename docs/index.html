🎯 What Is PostureIQ?

PostureIQ is an AI-powered compliance and security assessment tool that automatically audits your Azure subscriptions, Entra ID tenant, and Microsoft 365 environment against industry compliance frameworks. It collects evidence from 64 data sources, evaluates 525 controls across 11 frameworks (NIST 800-53, FedRAMP, CIS, PCI-DSS, ISO 27001, SOC 2, HIPAA, NIST CSF, CSA-CCM, MCSB, GDPR), and generates detailed reports with findings, scores, and remediation guidance.

It runs as a Microsoft Foundry AI Agent (conversational) or as standalone CLI scripts (headless for CI/CD). The agent understands natural language queries like "How compliant are we with NIST 800-53?" or "What are our top 10 risks?" and produces detailed, evidence-backed answers.

🔍 What Problems Does It Solve?

Manual compliance audits take weeks and go stale immediately. This tool automates the entire cycle — collect evidence, evaluate controls, generate reports — in minutes. Run it weekly in CI/CD to catch compliance drift before auditors do.

👥 Who Is It For?

Security engineers who need continuous compliance monitoring. Compliance officers who need audit-ready evidence. Platform teams who want security gates in deployment pipelines. CISOs who need risk dashboards.

🏗️ How Does It Work?

Three-phase pipeline: Collect (64 collectors gather evidence from Azure ARM, Microsoft Graph, Entra ID) → Evaluate (113 check functions across 10 security domains) → Report (24 generators produce HTML, JSON, Markdown, Excel, OSCAL, SARIF, PDF).

🤖 AI Agent vs CLI

The AI Agent runs on Microsoft Foundry with 14 registered tools — invoke via natural language. The CLI scripts (8 total) run headlessly for automation, CI/CD, and scheduled assessments with no human interaction.

🟢 L100 — Fundamentals BEGINNER

Start here if you're new to the tool, to compliance frameworks, or to Azure security. These documents explain what the tool does and how to get started.

What Is Compliance?

Compliance means your IT systems meet specific security standards required by regulations (HIPAA for healthcare, PCI-DSS for payments) or best practices (NIST, CIS). Each standard defines controls — specific security requirements like "enforce MFA" or "encrypt data at rest." This tool checks your Azure/Entra environment against those controls automatically.

What Is Azure + Entra ID?

Azure is Microsoft's cloud platform where your VMs, databases, storage, and services run. Entra ID (formerly Azure AD) is the identity platform that manages users, groups, app registrations, and access policies. This tool audits both — Azure resources via ARM APIs and Entra identity data via Microsoft Graph.

What Are Frameworks?

A compliance framework is a structured set of security controls. Examples: NIST 800-53 (83 controls, US government standard), PCI-DSS (51 controls, payment card industry), HIPAA (43 controls, healthcare). This tool maps controls from 11 frameworks to Azure/Entra evidence and evaluates each one.

How Do I Run It?

Two ways: (1) AI Agent — deploy to Microsoft Foundry and ask questions in natural language. (2) CLI — run any of the 8 CLI scripts (e.g., python run_risk_analysis.py --tenant YOUR-TENANT-ID) for a targeted assessment. Output goes to output/YYYYMMDD_HHMMSS/ with HTML, JSON, and Markdown reports.

🔵 L200 — Architecture & Concepts INTERMEDIATE

Understand the internal architecture, the three-phase pipeline (Collect → Evaluate → Report), evidence types, evaluator domains, and how compliance scoring works.

Three-Phase Pipeline

Phase 1: Collect — 64 collectors (49 Azure + 13 Entra + 2 standalone) gather evidence. Azure and Entra groups run concurrently. Phase 2: Evaluate — 113 check functions across 10 security domains evaluate 525 controls against gathered evidence. Phase 3: Report — 24 generators produce output in 8+ formats.

Evidence Types (218)

Each collector produces evidence records keyed by type (e.g., azure-keyvault, entra-conditional-access-policy). The engine builds an O(1) lookup index. Controls reference evidence types in their evidence_types field. If primary evidence is missing, compensating_evidence provides fallback.

Security Domains (10)

Controls are grouped into domains: access (8 handlers), identity (22), data_protection (22), network (16), logging (13), governance (21), incident_response (6), change_management (4), business_continuity (4), asset_management (4).

Severity-Weighted Scoring

Each finding carries a severity weight: critical=4, high=3, medium=2, low=1, informational=0. The compliance score is a weighted percentage with 50% partial credit. Framework-level and domain-level breakdowns are provided.

🟣 L300 — Operations PRACTITIONER

Day-to-day usage: configuration, CLI commands, report formats, CI/CD pipelines, suppression rules, and authentication modes.

4 Authentication Modes

auto (DefaultAzureCredential), serviceprincipal, appregistration, azurecli. Set via ENTERPRISESECURITYIQ_AUTH_MODE env var. Preflight check probes ARM + Graph + Entra to detect permission boundaries before collection.

8 CLI Scripts

run_rbac_report, run_query, run_risk_analysis, run_data_security, run_copilot_readiness, run_ai_agent_security, plus 2 determinism checks. Some support --fail-on-severity for CI/CD gates.

Report Formats

24 report generators produce: HTML (interactive), JSON (machine-readable), Markdown, Excel, OSCAL, SARIF (static analysis), PDF (via Playwright headless Chromium). Per-framework subfolders with drift reports when delta mode is active.

Suppression Rules

Mute accepted risks via suppressions.json with regex-based control_id and resource matching, ISO date expiry, and full audit trail. Suppressed findings stay in JSON exports for auditability.

🟠 L400 — Advanced Extension ADVANCED

Extend the tool: add custom compliance frameworks, write new collectors, create evaluator functions, and build plugins.

Add a Framework

Drop a JSON mapping file in AIAgent/app/postureiq_frameworks/ with control_id, domain, severity, evidence_types, and evaluation_logic fields. Register in the AVAILABLE_FRAMEWORKS dict. Controls are snake_case (not camelCase).

Add a Collector

Create a Python module in collectors/azure/ or collectors/entra/ with the @register_collector(name, plane, source, priority) decorator. Auto-discovered via pkgutil.iter_modules(). No manual import needed.

Add an Evaluator

Write a handler function with signature _check_xxx(control_id, control, evidence, evidence_index, thresholds) → list[dict]. Add to the domain dispatch dict. Reference the function name in framework JSON's evaluation_logic field.

Plugin System

Drop a .py file in plugins/ with an evaluate(control, evidence_index) → tuple[str, list[dict]] function. Set custom_evaluator field in framework JSON to the plugin name. Auto-discovered by load_plugins().

🔴 L500 — Expert Internals EXPERT

Deep internals: evaluation dispatch mechanics, determinism guarantees, checkpoint/resume implementation, error recovery, and troubleshooting.

Two-Stage Dispatch

The engine reads a control's domain field and dispatches to the matching domain evaluator. If all findings return not_assessed, the engine checks _CROSS_DOMAIN_MAP (28 routes) and redirects to an alternative domain. This ensures controls get evaluated even when primary evidence is in a different domain.

Determinism Guarantees

Three determinism check scripts verify evaluation reproducibility: collect once, evaluate N times (identical + shuffled order), compare SHA-256 hashes after stripping 12 volatile fields. Phase E checks report generation determinism. Exit code 0 = PASS, 1 = FAIL.

Collector Error Recovery

Base collector: 3 retries with exponential backoff (2s → 4s → 8s). AccessDeniedError creates a marker — pipeline continues. 429 reads Retry-After header. Timeouts preserve partial evidence via _partial_evidence attribute.

Checkpoint / Resume

.checkpoint.json stores {completed, failed, evidence}. On resume, completed collectors are skipped. Corrupt checkpoint → fresh start. Auto-deleted on success. .last_run.json enables delta/drift analysis.
