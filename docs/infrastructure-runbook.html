Architecture Overview

PostureIQ runs as a Docker container on Azure Container Apps, backed by Azure AI Foundry for LLM reasoning. The SPA authenticates users via MSAL.js, and all assessments run under the calling user's Entra permissions — no privilege escalation through the managed identity.

Platform Architecture
sequenceDiagram participant Browser as User Browser (SPA) participant CA as Container App
FastAPI + Uvicorn participant Foundry as Azure AI Foundry
gpt-4.1 / gpt-5.1 participant Graph as Microsoft Graph participant ARM as Azure ARM Browser->>Browser: MSAL.js login (PKCE) Browser->>CA: POST /chat + Bearer tokens (Graph + ARM) CA->>Foundry: Agent tool calls (MI token) Foundry-->>CA: LLM reasoning + function calls CA->>Graph: Delegated Graph queries (user token) Graph-->>CA: Entra / M365 data CA->>ARM: Delegated ARM queries (user token) ARM-->>CA: Azure resource data CA-->>Browser: SSE streamed response
  1. MSAL.js Login: SPA authenticates the user via OAuth 2.0 Authorization Code Flow with PKCE, obtaining delegated tokens for Graph and ARM.
  2. POST /chat: Browser sends the user's message along with graph_token and arm_token to the FastAPI backend.
  3. Agent Reasoning: The AI agent (powered by Azure AI Foundry) decides which tools to call based on the user's request.
  4. Delegated API Calls: All Graph and ARM calls use the user's tokens — assessments run under the user's own Entra permissions.
  5. SSE Streaming: Results stream back to the browser in real-time via Server-Sent Events.

Authentication Model

IdentityUsed ForToken Type
User (MSAL.js)All assessments — Graph + ARM API callsDelegated (user context)
ESIQNew-identity (MI)ACR pull, Foundry agent registration, infrastructure-onlyApplication (no user context)
ESIQNew-Dashboard (App Reg)SPA client — defines redirect URIs, declares delegated scopesN/A (template only)
Critical: The /chat endpoint requires user-delegated tokens. Requests without valid graph_token and arm_token are rejected with HTTP 401. Assessments always run under the calling user's Entra permissions — no privilege escalation through the managed identity.

Deployment Sequence — 16 Steps

The full deployment from zero to running platform. Can be executed as a single command via deploy.ps1, or step-by-step.

16-Step Deployment Sequence
sequenceDiagram participant Op as Operator participant PS as deploy.ps1 participant Az as Azure ARM participant Entra as Entra ID participant ACR as Container Registry participant CA as Container Apps Op->>PS: ./deploy.ps1 -BaseName ESIQNew rect rgb(30, 60, 90) Note over PS,Az: Phase 1 — Core Infrastructure (Steps 1–10) PS->>Az: Step 1: Create Resource Group PS->>Az: Step 2: Create Foundry Resource (AI Services) PS->>Az: Step 3: Create Foundry Project PS->>Az: Step 4: Deploy gpt-4.1 (30K TPM) PS->>Az: Step 5: Deploy gpt-5.1 (30K TPM) PS->>Az: Step 6: Create Storage Account (LRS, TLS 1.2) PS->>Az: Step 7: Create Key Vault (RBAC mode) PS->>Az: Step 8: Create Log Analytics Workspace PS->>Az: Step 9: Create Application Insights PS->>Az: Step 10: Create Container Registry (Basic) end rect rgb(30, 80, 50) Note over PS,Entra: Phase 2 — Identity & RBAC (Steps 11–12) PS->>Az: Step 11: Create User-Assigned Managed Identity PS->>Az: Step 12a: Assign 5 Azure RBAC roles PS->>Entra: Step 12b: Grant 8 Graph API application permissions end rect rgb(80, 50, 30) Note over PS,CA: Phase 3 — Build & Deploy (Steps 13–16) PS->>CA: Step 13: Create Container Apps Environment PS->>ACR: Step 14a: Build Docker image (ACR Tasks) PS->>CA: Step 14b: Create Container App (1 vCPU / 2 GiB) PS->>Entra: Step 15: Create App Registration (SPA + PKCE) PS->>ACR: Step 16: Patch SPA config → rebuild → update end PS-->>Op: Deployment complete — FQDN returned
  1. Phase 1 (Steps 1–10): Creates all Azure infrastructure — resource group, AI Foundry with custom domain + project management, model deployments, storage, monitoring stack, and container registry.
  2. Phase 2 (Steps 11–12): Creates the managed identity and assigns 5 Azure RBAC roles (AcrPull, Reader, Security Reader, Cognitive Services OpenAI User, Azure AI Developer) + 8 Microsoft Graph application permissions.
  3. Phase 3 (Steps 13–16): Creates the Container Apps Environment, builds the Docker image remotely via ACR Tasks, creates the Container App, registers the SPA in Entra ID, patches MSAL config in webapp/index.html, and redeploys.
One-Command Deployment: 
.\Infra-Foundary-New\deploy.ps1 -BaseName "ESIQNew" -Location "swedencentral" -SubscriptionName "AI"

Day-2 Operations

Common operational tasks: redeploy code changes, view logs, scale, update environment variables, rotate model deployments. 

# Quick Redeploy (code changes only)
.\Infra-Foundary-New\redeploy-image.ps1 -Tag "v34-perms"

# Manual redeploy
az acr build --registry esiqnewacr --image esiqnew-agent:v34-perms \
    --file AIAgent/Dockerfile . --no-logs \
    --build-arg CACHEBUST=$(Get-Date -Format 'yyyyMMddHHmmss')
az containerapp update --name esiqnew-agent --resource-group ESIQNew-RG \
    --image esiqnewacr.azurecr.io/esiqnew-agent:v34-perms -o table

# View logs (real-time)
az containerapp logs show --name esiqnew-agent --resource-group ESIQNew-RG --follow

# Scale configuration
az containerapp update --name esiqnew-agent --resource-group ESIQNew-RG \
    --min-replicas 1 --max-replicas 5

# Update environment variables
az containerapp update --name esiqnew-agent --resource-group ESIQNew-RG \
    --set-env-vars "NEW_VAR=value"
Important: Always use a NEW tag to force Container Apps to re-pull the image. Reusing a tag (e.g. :latest) may serve a cached version.

Disaster Recovery & Rollback

Rollback to a previous image version, reactivate a previous revision, or fully rebuild from scratch. 

# Rollback to previous image version
az acr repository show-tags --name esiqnewacr --repository esiqnew-agent -o table
az containerapp update --name esiqnew-agent --resource-group ESIQNew-RG \
    --image esiqnewacr.azurecr.io/esiqnew-agent:v33-session -o table

# Rollback to previous revision
az containerapp revision list --name esiqnew-agent --resource-group ESIQNew-RG -o table
az containerapp revision activate --name esiqnew-agent --resource-group ESIQNew-RG \
    --revision {revision-name}

# Full disaster recovery (RG deleted)
.\Infra-Foundary-New\deploy.ps1 -BaseName "ESIQNew" -Location "swedencentral" -SubscriptionName "AI"
# FQDN will change → update App Registration redirect URIs
Data Recovery: Assessment reports are stored locally in /agent/output inside the container and are not persistent across container restarts. Download reports from the SPA immediately after completion.

Teardown / Decommission

WARNING: This is destructive and irreversible. All resources, data, and configurations will be permanently deleted. 
# Delete entire resource group (removes all 13 Azure resources)
az group delete --name ESIQNew-RG --yes --no-wait

# Also clean up Entra objects
az ad app delete --id $(az ad app list --display-name ESIQNew-Dashboard --query "[0].id" -o tsv)

# Verify deletion
az group exists --name ESIQNew-RG -o tsv   # should return "false"

Not cleaned up by RG deletion: Entra App Registration (delete separately). Directory role assignments and Graph permissions on the MI are auto-removed when the MI is deleted.
