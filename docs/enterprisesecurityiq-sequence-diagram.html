6. Detailed Phase Explanation

0 Phase 0 — CLI Bootstrap & Pre-Flight Validation

The assessment begins when the user invokes run_assessment.py via the command line. The CLI performs three critical setup steps before any Azure API calls:

  • Argument parsing: argparse extracts --tenant, --framework (validated against 11 supported: NIST-800-53, CIS, HIPAA, PCI-DSS, FedRAMP, SOC2, ISO-27001, MCSB, NIST-CSF, CSA-CCM, GDPR), --subscriptions filter, and --output-dir.
  • Configuration loading: AssessmentConfig.from_env() loads settings from a JSON config file (if ENTERPRISESECURITYIQ_CONFIG env var is set), then overlays environment variable overrides. Key settings include azure_batch_size=4, entra_batch_size=3, collector_timeout=600s, checkpoint_enabled=true, and compliance thresholds.
  • Pre-flight validation: ComplianceCredentials.preflight_check() performs four API probes — ARM subscription listing, Graph /me for user identity, /organization for tenant info, and /me/memberOf to verify the caller holds at least one of five sufficient Entra roles (Global Admin, Global Reader, Security Admin, Security Reader, Compliance Admin). If any probe fails, the assessment is aborted with a clear error before expensive collection runs.

1 Phase 1 — Evidence Collection (56 Collectors)

The orchestrator coordinates parallel execution of 44 Azure collectors and 12 Entra collectors, each producing standardised EvidenceRecord dictionaries with PascalCase keys.

  • Collector discovery: At module-load time, registry.py uses pkgutil.iter_modules() to auto-discover all Python modules under collectors/azure/ and collectors/entra/. Each module's @register_collector decorator populates a central registry with the collector function, name, source (azure/entra), plane (control/data), and priority (execution order).
  • Resource inventory pre-population: Before collectors run, the ResourceInventory singleton pre-fetches all resources across all subscriptions via ARM resources.list(), creating an in-memory index by resource type and subscription ID. This avoids N+1 API calls across collectors.
  • Batched parallel execution: Azure collectors run in batches of 4 (configurable) using asyncio.gather(). Each collector is wrapped in asyncio.wait_for(timeout=600s) for safety. Entra collectors run similarly in batches of 3. After each batch, a checkpoint is saved to .checkpoint.json with all collected evidence and the list of completed/failed collectors, enabling resume on interruption.
  • ARM pagination: paginate_arm(pager) handles three SDK return patterns — async iterators (standard pagers), coroutines (some SDK methods), and plain iterables. It raises AccessDeniedError on 403/401 responses.
  • Graph pagination: paginate_graph(request_builder, top=999) follows odata.nextLink for multi-page Graph responses, handles 429 throttling by parsing Retry-After headers, and raises AccessDeniedError on 403/401.
  • Evidence production: Each collector calls make_evidence(source, collector, evidence_type, description, data, resource_id, resource_type) which creates an EvidenceRecord dataclass with a UUID, UTC timestamp, and PascalCase-serialised fields.

2 Phase 2 — Compliance Evaluation & Scoring

The evaluation engine maps collected evidence to framework controls, dispatches to domain-specific evaluators, and produces severity-weighted compliance scores.

  • Evidence indexing: _index_evidence() builds a dictionary keyed by EvidenceType (e.g. "azure-role-assignment") pointing to lists of matching records. This enables O(1) evidence lookup per control.
  • Framework mapping: For each selected framework, _load_mappings() reads its JSON file from app/frameworks/. Each control definition specifies a domain (routing key), evidence_types (lookup keys), evaluation_logic (function name), severity (scoring weight), and optionally compensating_evidence (fallback types).
  • Domain dispatch: 10 domain evaluators (access, identity, data_protection, logging, network, governance, compute, database, application, encryption) are registered in DOMAIN_EVALUATORS. The engine dispatches each control to its domain evaluator, passing the control definition, matching evidence, the full evidence index, and threshold configuration. Each evaluator executes the specific evaluation_logic function (e.g. _check_account_management()) and returns a list of FindingRecord dictionaries.
  • Cross-domain fallback: If a primary domain evaluator returns all not_assessed findings, the engine consults _CROSS_DOMAIN_MAP to try an alternative domain's evaluator. This handles controls that span multiple security domains.
  • Scoring: _weighted_score() assigns weights by severity (critical=4, high=3, medium=2, low=1), grants 50% credit for partial compliance, and excludes missing_evidence/not_assessed from the denominator to avoid unfairly penalising limited-permission assessments. _maturity_level() computes a composite score: pass_ratio×50 + critical_pass×30 + depth_ratio×20 mapping to a 5-level maturity scale (Initial → Developing → Defined → Managed → Optimized).
  • Suppression system: Post-evaluation, configurable suppression rules can filter out known-accepted findings (e.g. accepted risks, compensating controls in place).

3 Phase 3 — Report Generation (24 Generators)

The orchestrator invokes up to 24 report generators, each wrapped in _safe() for isolated error handling — a single report failure never blocks others.

  • Per-framework reports: When multiple frameworks are assessed, each gets its own subfolder (output/{ts}/NIST-800-53/, output/{ts}/CIS/, etc.) containing compliance-report.html (full sidebar-nav HTML report with inventory, RBAC analysis, control details, findings, and remediation roadmap), compliance-report.md (Markdown version), and master-report.html/.md/.xlsx (flat finding table, sortable and searchable).
  • Shared reports: Generated once in the root output directory — methodology-report.html (4-phase pipeline diagram, collector stats), executive-dashboard.html (score gauge, framework comparison cards, top-10 findings, domain heatmap), drift-report.html (if a prior assessment exists, shows control status changes).
  • Data exports: control-results.csv/json, findings.csv/json, missing-evidence.csv/json, compliance-data.json (full serialised assessment data for downstream consumption).
  • Machine-readable formats: SARIF export for CI/CD pipeline integration (GitHub, Azure DevOps), OSCAL export for federal compliance workflows.
  • Raw evidence archive: All evidence records are grouped by EvidenceType and saved as individual JSON files in raw/ — providing a full audit trail for independent verification.
  • Remediation playbooks: Auto-generated per-finding-type remediation guides in remediation/ with Azure CLI, PowerShell, and Bicep code snippets.

4 Phase 4 — Cleanup & Results

The orchestrator performs final cleanup: deletes the .checkpoint.json file (no longer needed after successful completion), saves last-run metadata for future delta comparisons, and returns the complete results dictionary to the CLI. The CLI prints a summary table with compliance scores, finding counts by severity, and the output directory path.

8. Data Flow & Data Structures

Evidence Record (collection output)

Each collector produces a list of EvidenceRecord dictionaries (PascalCase-serialised via to_dict()). Key fields:

  • EvidenceId — UUID v4, unique per record
  • Source — "Azure" or "Entra"
  • Collector — e.g. "AzureRBAC", "EntraConditionalAccess"
  • EvidenceType — the key lookup field used by the evaluation engine index (e.g. "azure-role-assignment", "entra-conditional-access-policy") — over 120 distinct types
  • Data — nested dict with resource-specific fields (varies per collector)
  • ResourceId, ResourceType — ARM resource ID and type
  • CollectedAt — ISO 8601 UTC timestamp

Evidence Index (evaluation input)

_index_evidence() transforms the flat evidence list into a dictionary keyed by EvidenceType → list of matching records. This provides O(1) lookup when the evaluation engine needs to gather evidence for a specific control. Example: index["azure-nsg-rule"] instantly returns all NSG rule evidence records without scanning the full list.

Finding Record (evaluation output)

Each domain evaluator produces FindingRecord dictionaries. Key fields:

  • FindingId — UUID v4
  • ControlId, Framework, ControlTitle — control identification
  • Status — compliant, non_compliant, partial, missing_evidence, or not_assessed
  • Severity — critical, high, medium, low, or informational
  • Domain — evaluation domain (access, network, etc.)
  • Description — human-readable finding text
  • Recommendation — remediation guidance
  • SupportingEvidence — array of {ResourceId, ResourceName, ResourceType} references

Compliance Summary (aggregated output)

The final summary structure contains:

  • ComplianceScore — severity-weighted percentage (critical=4×, high=3×, medium=2×, low=1×; partial=50% credit; missing/not_assessed excluded from denominator)
  • FrameworkSummaries — per-framework breakdown of control counts by status
  • DomainScores — per-domain scores with maturity levels
  • OverallMaturity — 5-level scale: Initial (0-30) → Developing (30-50) → Defined (50-70) → Managed (70-85) → Optimized (85-100)
Data transformation pipeline: ARM/Graph API responses → Collector SDK objects → make_evidence() → EvidenceRecord[] → _index_evidence() → evidence_index{} → Domain Evaluator → FindingRecord[] → control aggregation → ComplianceControlResult[] → scoring → Summary → Report Generators → HTML/MD/XLSX/CSV/JSON/SARIF/OSCAL files

9. Operational Modes

▶ Mode 1: Full Compliance Assessment (run_assessment.py)

End-to-end assessment: pre-flight → collect all evidence (56 collectors) → evaluate against N frameworks → generate all reports. This is the primary mode depicted in the sequence diagrams above.

python run_assessment.py --tenant T --framework NIST-800-53 CIS HIPAA PCI-DSS

▶ Mode 2: RBAC Hierarchy Report (run_rbac_tree.py)

Standalone visualisation of the full Azure RBAC hierarchy: Management Groups → Subscriptions → Resource Groups → Resources, with role assignments, PIM eligibility, principal resolution, and group member expansion. Does not use the standard collector registry or evaluation engine. Produces an interactive HTML tree report and raw JSON.

python run_rbac_tree.py --tenant T [--subscriptions sub1 sub2]

▶ Mode 3: Interactive Query REPL (run_query.py)

On-demand search of Azure and Entra resources without a full assessment. Supports natural language queries (dispatched to Azure Resource Graph or Graph API), raw KQL via /arg, user detail lookups via /user, and compliance findings search via /findings (requires a prior assessment run). Results can be exported to JSON.

python run_query.py --tenant T [--query "list all VMs without encryption"]

▶ Mode 4: Risk Analysis (run_risk_analysis.py)

Security risk gap analysis across 4 categories (identity, network, defender, configuration). Can reuse evidence from a prior assessment via --evidence flag or collect fresh data. Produces risk scores per category.

python run_risk_analysis.py --tenant T [--evidence path] [--category identity,network]

▶ Mode 5: Data Security Assessment (run_data_security.py)

Data-layer security posture across 5 categories (storage, database, keyvault, encryption, classification). Focuses on data-at-rest encryption, access controls, key management, and data classification policies.

python run_data_security.py --tenant T [--category storage,encryption]

▶ Mode 6: Foundry Agent (main.py + agent.py)

Azure AI Foundry hosted agent mode exposing 5 tools (run_assessment, query_results, search_tenant, analyze_risk, assess_data_security) via the responses v1 protocol. Uses Managed Identity authentication and an LLM (configurable, default gpt-4.1) for conversational compliance analysis.

▶ Mode 7: REST API (api.py)

FastAPI server exposing POST /assessments (async assessment with BackgroundTasks, returns task_id), GET /assessments/{task_id} (poll for results), and GET /health. Supports webhook notification on completion.

uvicorn app.api:app --host 0.0.0.0 --port 8000
