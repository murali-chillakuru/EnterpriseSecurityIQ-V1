Scoring Algorithm & Dispatch

The engine uses severity-weighted scoring to aggregate compliance results. Each assessed control contributes based on its severity weight.

sequenceDiagram participant FW as Framework JSON participant ENG as engine.py participant DOM as Domain Evaluator participant XD as Cross-Domain Map participant SUP as Suppressions participant PLG as Plugins participant SC as Score Aggregator FW->>ENG: Control with evaluation_logic ENG->>DOM: Route to primary domain alt Domain handles it DOM-->>ENG: (status, findings) else Not handled ENG->>XD: Lookup _CROSS_DOMAIN_MAP (28 routes) XD-->>ENG: Redirected domain evaluator ENG->>DOM: Route to fallback domain DOM-->>ENG: (status, findings) end ENG->>SUP: Check suppression rules alt Suppressed SUP-->>ENG: Skip finding (logged) else Not suppressed SUP-->>ENG: Finding passes through end ENG->>PLG: Check custom_evaluator tag alt Plugin exists PLG-->>ENG: Plugin result end ENG->>SC: Add weighted result Note over SC: critical=4, high=3, medium=2, low=1 SC-->>ENG: Updated domain & overall scores

Sequence Walkthrough

  1. Framework JSON provides each control's evaluation_logic function name and required evidence_types
  2. engine.py routes to the primary domain evaluator (access, identity, data_protection, etc.)
  3. If the domain doesn't handle the function, the Cross-Domain Map redirects to one of 28 pre-configured fallback routes
  4. Suppressions are checked — matching rules (control_id glob + resource regex) skip findings
  5. Plugins run for controls tagged custom_evaluator, injecting external evaluation logic
  6. Score Aggregator applies severity weights: compliant contributes full weight, partial contributes half, non-compliant contributes zero. missing_evidence and not_assessed are excluded from the denominator

Severity Weights

SeverityWeightScore Impact
Critical4Heaviest impact on compliance percentage
High3Significant impact
Medium2Moderate impact

Score Formula

weighted_pass  = Σ weight[severity]  for each compliant control
weighted_total = Σ weight[severity]  for each assessed control
compliance_pct = (weighted_pass / weighted_total) × 100

partial status contributes half the severity weight. missing_evidence / not_assessed are excluded from the denominator.

Fallback Dispatch Explained

  1. engine.py first tries the domain specified in the control's evaluation_logic field
  2. If that domain evaluator doesn't have a handler for the function, the engine consults _CROSS_DOMAIN_MAP
  3. The map contains 28 redirects — for example, check_sentinel_monitoring routes to Incident Response even when invoked from a Logging-domain control
  4. If no route matches and no domain handles it, a generic fallback checks evidence presence → partial if present, missing_evidence if absent

Suppression Rules

The suppression engine (evaluators/suppressions.py) allows administrators to skip or downgrade specific findings.

Rule Structure

FieldTypeDescription
control_idstr | nullExact match or glob pattern on control ID (e.g., "AC-*")
resourcestr | nullRegex pattern matching resource name/ID
reasonstrRequired justification for suppression
expiresstr | nullISO date (YYYY-MM-DD); suppression ignored after expiry

Matching Logic

  1. Rules are tested in order of definition
  2. A rule matches when both control_id (exact/glob) AND resource (regex) match
  3. Expired rules (past expires date) are skipped with a warning log
  4. Rules expiring within 7 days trigger a warning about upcoming expiration
  5. Matched rules suppress the finding from the final report but are tracked in metadata

Suppression Flow

sequenceDiagram participant F as Finding participant S as suppressions.py participant R as Rules List participant O as Output F->>S: Check finding against rules loop Each suppression rule S->>R: Load rule (control_id, resource, expires) alt Expired R-->>S: Skip rule (log warning) else Expiring within 7 days R-->>S: Apply rule + warn else Active alt control_id glob matches AND resource regex matches R-->>S: MATCH — suppress finding S-->>O: Finding suppressed (logged in metadata) else No match R-->>S: Continue to next rule end end end S-->>O: Finding passes through (not suppressed)

Suppression Flow Explained

  1. Each finding is checked against the ordered list of suppression rules
  2. Expired rules are skipped and logged as warnings
  3. Rules nearing expiry (within 7 days) still apply but emit a warning
  4. Both control_id (glob match) and resource (regex match) must match for suppression
  5. Suppressed findings are excluded from reports but tracked in metadata for audit purposes

Example Rule

{
  "control_id": "AC-6",
  "resource": ".*-dev-.*",
  "reason": "Dev subscription excluded from least-privilege audit",
  "expires": "2025-12-31"
}

Plugin System

The plugin engine (evaluators/plugins.py) allows custom evaluation functions to be injected at runtime.

How It Works

  1. load_plugins(plugin_dir) scans a directory for .py files (skipping _-prefixed), dynamically imports each via importlib.util
  2. Each module must expose an evaluate() callable at module level
  3. run_plugins() iterates all controls tagged with custom_evaluator matching a plugin name

Plugin Signature

def evaluate(control: dict, evidence_index: dict[str, list]) -> tuple[str, list[dict]]:
    """
    Args:
        control: Framework control definition dict
        evidence_index: Dict mapping evidence_type → list of evidence records
    Returns:
        (status, findings) — status is one of the 5 compliance statuses
    """

Evidence Types

Each control specifies required evidence_types[] in its framework mapping JSON. Missing types result in missing_evidence status. Controls may also specify compensating_evidence — alternative evidence types that satisfy the requirement if primary evidence is unavailable (e.g., due to 403 permissions).

See Architecture for the full collector → evidence type mapping covering 200+ evidence types across 68 collectors.
