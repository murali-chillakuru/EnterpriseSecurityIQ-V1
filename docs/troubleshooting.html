🔐 Authentication Issues

Credential Resolution Order

sequenceDiagram participant App as Assessment Engine participant DAC as DefaultAzureCredential participant Env as Environment Variables participant MI as Managed Identity participant CLI as Azure CLI Cache participant AR as App Registration App->>DAC: authenticate() DAC->>Env: AZURE_CLIENT_ID + SECRET? alt Env vars present Env-->>DAC: ClientSecretCredential ✅ else No env vars DAC->>MI: Managed identity endpoint? alt Running in Azure MI-->>DAC: ManagedIdentityCredential ✅ else Not in Azure DAC->>CLI: az login cache? alt CLI authenticated CLI-->>DAC: AzureCliCredential ✅ else No CLI session DAC-->>App: CredentialUnavailableError ❌ end end end Note right of DAC: If AUTH_MODE=appregistration,
uses dedicated app creds
How credential resolution works:
  1. Environment Variables — Checked first. If AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_CLIENT_SECRET are all set, a ClientSecretCredential is created.
  2. Managed Identity — If running in Azure (Container Apps, App Service, VM), the managed identity endpoint is probed.
  3. Azure CLI — Falls back to the cached az login session. Ensure it hasn't expired.
  4. App Registration — When AUTH_MODE=appregistration, uses ENTERPRISESECURITYIQ_APP_CLIENT_ID + _SECRET instead.
"DefaultAzureCredential failed" 
az login --tenant <TENANT_ID>
az account show --query '{tenant:tenantId, subscription:name}'

🔍 Preflight Check Failures

Preflight Probe Sequence

sequenceDiagram participant C as ComplianceCredentials participant ARM as ARM API participant G1 as Graph /users participant G2 as Graph /conditionalAccess participant G3 as Graph /roleManagement C->>ARM: List subscriptions ARM-->>C: ✅ subscriptions list C->>G1: GET /users?$top=1 alt 200 OK G1-->>C: ✅ User.Read.All confirmed else 403 Forbidden G1-->>C: ⚠️ Warning — reduced scope else 401 Unauthorized G1-->>C: ❌ Blocking — credential invalid end C->>G2: GET /identity/conditionalAccess/policies alt 200 OK G2-->>C: ✅ Policy.Read.All confirmed else 403 G2-->>C: ⚠️ Warning — CA data unavailable end C->>G3: GET /roleManagement/directory/roleAssignments alt 200 OK G3-->>C: ✅ RoleManagement.Read.All confirmed else 403 G3-->>C: ⚠️ Warning — role data unavailable end
Preflight probes 3 Graph endpoints before collection begins:
  1. Users — Tests User.Read.All. A 403 means user-related collectors will produce access-denied markers.
  2. Conditional Access — Tests Policy.Read.All. A 403 means CA policy data will be unavailable.
  3. Role Management — Tests RoleManagement.Read.All. A 403 means directory role data is missing.

403 = warning, assessment continues with reduced scope. 401 = blocking error, assessment stops.

⚠️ 401 Errors During Preflight If you get a 401 during preflight, the credential is not valid for Graph access. Verify with: 
az rest --method GET --url "https://graph.microsoft.com/v1.0/me"

📡 Collector Failures

Collector Error Handling Flow

flowchart TD A[Collector Invoked] --> B{API Call} B -->|200 OK| C[Return Evidence] B -->|401/403| D[AccessDeniedError] B -->|429| E[Read Retry-After Header] B -->|500+| F{Retry Count < 3?} B -->|Timeout| G{Has _partial_evidence?} D --> H[Create access-denied marker] H --> I[success=True, pipeline continues] E --> J[Wait Retry-After seconds] J --> B F -->|Yes| K[Exponential Backoff] K --> B F -->|No| L[Hard Failure: success=False] G -->|Yes| M[Return partial data] G -->|No| L C --> N[Collector Complete ✅] I --> N M --> N L --> O[Collector Failed ❌] style C fill:#3FB950,color:#fff style I fill:#D29922,color:#fff style M fill:#39D2C0,color:#fff style L fill:#F85149,color:#fff
Three possible outcomes for every collector:
  1. Success (green) — Evidence records returned normally.
  2. Access Denied (orange) — 401/403 creates an access-denied marker. Treated as success — the pipeline continues and affected controls show missing_evidence.
  3. Hard Failure (red) — After 3 retries with exponential backoff, the collector is marked as failed. 429 responses use Retry-After header for backoff timing.
  4. Partial Recovery (cyan) — If a collector times out but has a _partial_evidence attribute, partial results are preserved.
Timeout Errors

Log pattern: WARNING: <collector> collector timed out after <N>s

Code default: 600s. Shipped config: 300s. Fix:

{ "collectors": { "collectorTimeout": 900 } }
Module Import Failures

Log pattern: WARNING: Failed to import collector module <name>

If a collector module has a syntax error or missing dependency, it's skipped during discover_collectors(). The pipeline continues with a reduced collector count.

Large Tenant Optimization 
{
  "collectors": {
    "subscriptionFilter": ["sub-id-1", "sub-id-2"],
    "userSampleLimit": 5000,
    "azureBatchSize": 4,
    "entraBatchSize": 3,
    "collectorTimeout": 600
  }
}

Each evidence record adds ~1-2 KB. Monitor memory usage on large tenants.

💾 Checkpoint & Resume

Checkpoint Lifecycle

sequenceDiagram participant O as Orchestrator participant FS as .checkpoint.json participant C as Collectors O->>FS: Check for existing checkpoint alt Checkpoint exists FS-->>O: Load evidence + completed set Note over O: Skip already-completed collectors else No checkpoint Note over O: Start fresh end loop For each collector batch O->>C: Run collectors C-->>O: Evidence records O->>FS: Save checkpoint (evidence + completed + failed) end alt All collectors complete O->>FS: Delete .checkpoint.json ✅ else Interrupted Note over FS: .checkpoint.json preserved for resume end
Checkpoint lifecycle:
  1. Check — On startup, the orchestrator looks for .checkpoint.json in the output directory.
  2. Resume — If found, loads evidence and the completed collector set. Already-completed collectors are skipped.
  3. Save — After each batch, the checkpoint is updated with current state.
  4. Cleanup — After successful collection, the checkpoint file is deleted. If it persists, the last run didn't complete.
🔄

Resume After Failure

Re-run the same command. Completed collectors are skipped automatically.

# Same command resumes
python run_assessment.py \
  --tenant "..." --framework FedRAMP
⚠️

Corrupt Checkpoint

If .checkpoint.json is malformed, the engine logs a warning and starts fresh — it does NOT crash.

WARNING: Checkpoint corrupt, starting fresh run

📊

Delta Mode

The orchestrator also supports delta/incremental mode via .last_run.json, tracking resource IDs and timestamps for change detection.

Cross-Domain Fallback Surprises

The evaluation engine has 28 cross-domain fallback routes. If a primary evaluator lacks evidence, the _CROSS_DOMAIN_MAP redirects to an alternative domain. Enable DEBUG logging to trace:

DEBUG: Cross-domain fallback: governance → access (control AC-2)
Suppressions Hiding Findings

If expected findings are missing, check for active suppression rules. Suppressions use control ID glob patterns and resource regex matching with expiry dates. See Suppressions Guide.

📄 Report Generation Issues

✅ Fault Isolation Report generation is wrapped in _safe() error handling — one failing report won't block others. Each report type is generated independently.
Empty or Missing Reports

Log pattern: ERROR: Report '<type>' failed for <framework>: <error>

Common causes: missing evidence, framework mapping not found, template rendering error.

Excel Report Errors 
pip install openpyxl
Multi-Tenant Report Failures

Log pattern: ERROR: Tenant <id> failed: <error>

The engine continues to the next tenant — per-tenant errors are non-blocking.

📑 PDF Export (Playwright)

PDF generation uses Playwright with headless Chromium. It's always optional — failures are logged but never crash the assessment.

🌐

Chromium Not Installed Common

WARNING: [PDF] Chromium launch failed

pip install playwright
playwright install chromium
📄

Single File Failure

WARNING: [PDF] Failed to convert <file>

Remaining HTML files continue to be converted.

🔍

HTML Not Found

WARNING: [PDF] HTML file not found

The report generator likely failed silently for that report type.

📦

Batch Mode Fallback

convert_all_html_to_pdf() reuses one browser. If batch fails, falls back to individual html_to_pdf() calls.
