Overview

The Query Engine (query_engine.py + run_query.py) provides an interactive query interface for exploring Azure infrastructure, Entra ID identities, and assessment findings. It combines Azure Resource Graph KQL templates with Microsoft Graph Entra queries and full-text evidence search, all unified behind a natural language dispatch layer with keyword routing. Users can drill down into specific resources or identities with dedicated detail functions.

Key Capabilities: 11 pre-built ARG KQL templates for common infrastructure queries · 6 Entra ID query functions covering users, groups, apps, service principals, directory roles, and conditional access policies · Natural language dispatch with keyword-based routing · Full-text evidence and findings search with advanced filters · Interactive REPL with command history and export · Single-shot CLI mode for CI/CD integration.
⚡

Architecture

The query engine follows a dispatch-based architecture. User input enters the REPL loop or single-shot CLI, is routed through the natural language dispatcher to the appropriate backend (ARG, Graph, or findings store), and results are formatted and returned as structured table output.

Query Engine — Dispatch Flow
sequenceDiagram participant U as User participant R as run_query.py (REPL) participant D as dispatch_natural_language() participant ARG as Azure Resource Graph participant G as Microsoft Graph participant F as Findings / Evidence U->>R: "Show me public storage accounts" R->>D: dispatch_natural_language(creds, question) D->>D: Match keywords to query type alt ARG Query D->>ARG: query_resource_graph(kql) ARG-->>D: Resource results else Entra Query D->>G: query_entra_users/groups/apps G-->>D: Identity results else Findings Search D->>F: cross_reference_findings(query) F-->>D: Matching findings end D-->>R: {source, query_used, results, count} R-->>U: Formatted table output
  1. User Input — A natural language question or REPL command enters the system via run_query.py.
  2. NL Dispatch — dispatch_natural_language() scans keywords to classify the query as ARG, Entra, or findings-related.
  3. Backend Execution — The matched query type routes to Azure Resource Graph KQL, Microsoft Graph Entra API, or the local findings/evidence store.
  4. Result Formatting — Results are returned as a structured dict with source, query_used, results, and count fields, then rendered as a formatted table.

The engine ships with 11 pre-built Azure Resource Graph KQL templates covering the most common infrastructure queries. Each template is a parameterized KQL string executed via query_resource_graph().

keyvaultsEnumerate Key Vaults with soft-delete, purge-protection, and network ACL settings

resources_by_locationAggregate resource counts grouped by Azure region/location
Custom KQL: Beyond pre-built templates, the REPL supports /arg <KQL> for executing arbitrary KQL queries directly against Azure Resource Graph.

Entra Queries

Six dedicated functions query Microsoft Graph for Entra ID identity data. Each function handles pagination, throttling, and result normalization. Two additional drill-down functions provide detailed views of individual resources and users.

FunctionDescription
Core Entra Query Functions
query_entra_usersList all users with sign-in status, MFA registration, license assignments, and last sign-in timestamps
query_entra_groupsList all groups with membership type (dynamic/assigned), mail-enabled status, and security classification
query_entra_appsList application registrations with credential expiry dates, API permissions, and owner assignments
query_entra_service_principalsList service principals with credential status, app role assignments, and sign-in activity
query_entra_directory_rolesList activated directory roles with member counts and privileged role identification
query_entra_conditional_accessList conditional access policies with state (enabled/disabled/report-only), grant/session controls, and target assignments
Drill-Down Functions
get_resource_detailFetch full ARM properties for a single resource by resource ID, including sub-resources and configuration
get_entra_user_detailFetch comprehensive user profile: group memberships, app role assignments, authentication methods, sign-in logs

Evidence Search

The evidence search system provides two modes: basic search (free-text keyword matching across all evidence records) and advanced search (structured filters for precise targeting). The advanced search accepts a filters dictionary with any combination of the following keys.

Filter KeyTypeDescription
querystringFree-text search across finding title, description, and remediation fields
evidence_typestringFilter by evidence type (e.g., storage_accounts, nsg_rules, key_vaults)
resource_typestringFilter by ARM resource type (e.g., microsoft.storage/storageaccounts)
subscription_idstringFilter evidence to a specific Azure subscription
locationstringFilter by Azure region (e.g., eastus, westeurope)
collectorstringFilter by the collector function that gathered the evidence
has_fieldstringMatch evidence records containing a specific field name
field_valuestringMatch evidence records where the has_field contains this value
Search Modes: Basic search is ideal for quick keyword lookups (e.g., "public access"). Advanced search with structured filters is designed for precise, repeatable queries in CI/CD pipelines or compliance reporting workflows.

REPL Commands

The interactive REPL (run_query.py) provides a command-line interface with 9 commands. Natural language input is passed directly to the NL dispatcher, while slash-commands provide direct access to specific query backends.

CommandDescription
natural languageType any question — routed through dispatch_natural_language() to the best-matching backend
/arg <KQL>Execute a raw KQL query directly against Azure Resource Graph
/user <id>Fetch detailed Entra ID user profile via get_entra_user_detail()
/resource <id>Fetch full ARM resource detail via get_resource_detail()
/templatesList all 11 available ARG query templates with descriptions
/template <name>Execute a specific named ARG template (e.g., /template public_ips)
/findings <keyword>Search findings and evidence by keyword via cross_reference_findings()
/export <file.json>Export the last query results to a JSON file
/quitExit the interactive REPL session
⌨️

CLI Usage

The query engine supports both interactive REPL mode and single-shot CLI mode for scripting and CI/CD integration. 

# Interactive REPL mode
python run_query.py --interactive

# Single-shot natural language query
python run_query.py --query "Show me all public storage accounts"

# Raw ARG KQL query
python run_query.py --arg "Resources | where type == 'microsoft.storage/storageaccounts' | project name, location"

# Query with findings cross-reference
python run_query.py --query "non-compliant key vaults" --include-findings
